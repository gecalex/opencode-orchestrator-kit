## When to Use

Используй этот скилл:

- ✅ После /speckit.specify — перед /speckit.plan
- ✅ Когда спецификации неполные
- ✅ Когда есть противоречия
- ❌ НЕ используй сразу после создания спецификаций (только при необходимости)

## Назначение

Уточнить спецификации перед переходом к планированию:
- Выявить неочевидные сценарии
- Проверить полноту требований
- Уточнить неоднозначные формулировки

## Инструкции

### Шаг 1: Прочитать спецификации

Проверь все локации:

```bash
# 1. Speckit
find .specify/specs -name "*.md"

# 2. utA
find SPEC/specs -name "*.md"

# 3. Корень
find specs -name "*.md"
```

### Шаг 2: Анализ на неполноту

Проверь для каждой спецификации:

- [ ] Есть ID модуля?
- [ ] Есть описание функциональности?
- [ ] Есть входные данные?
- [ ] Есть выходные данные?
- [ ] Есть обработка ошибок?
- [ ] Есть ограничения?
- [ ] Есть зависимости от других модулей?

### Шаг 3: Анализ на противоречия

- [ ] Нет ли конфликтов между модулями?
- [ ] Нет ли дублирования функциональности?
- [ ] Нет ли конфликтов в интерфейсах?

### Шаг 4: Создание отчёта уточнений

Создай файл `.specify/specs/clarifications.md`:

```markdown
# Уточнения спецификаций

## Неполные области

### module-name
- Проблема: ...
- Вопрос: ...

## Противоречия

### module-name vs module-name
- Проблема: ...
- Предложение: ...

## Дополнительные сценарии

### module-name
- Сценарий: ...
```

## Результат

```json
{
  "status": "clarified",
  "incomplete": ["module-1", "module-2"],
  "conflicts": ["module-a vs module-b"],
  "clarification_file": ".specify/specs/clarifications.md"
}
```
