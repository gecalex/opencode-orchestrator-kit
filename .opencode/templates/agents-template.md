## 🚫 Блокирующие правила (ОБЯЗАТЕЛЬНО ДЛЯ ВСЕХ АГЕНТОВ)

### Аварийный тормоз
Перед запуском любого агента оркестратор проверяет:
- Соответствие текущего состояния workflow разрешенному для данного типа агента
- Наличие явного подтверждения пользователя на критических переходах
- Выполнение всех pre-condition проверок для текущей фазы
- Доступность необходимых MCP серверов (если требуются для задачи)

### Правило пре-условий
Каждая операция агента начинается с автоматической проверки пре-условий:
- Для аналитических агентов: доступность исходных данных и спецификаций
- Для кодеров: наличие написанных и прошедших тестов (TDD First!)
- Для рецензентов: наличие артефакта для рецензирования и понимания требований
- Для тестировщиков: наличие тестовых сценариев и среды выполнения

### Правило подтверждения пользователя
На следующих переходах требуется явное подтверждение пользователя:
- Переход от анализа к планированию (фаза 3→4)
- Переход от планирования к выполнению (фаза 6→7)
- Переход от написания тестов к написанию кода (фаза 9→10)
- Любое изменение критической архитектуры или базы данных
- Удаление файлов или компонентов системы

## 🚀 Быстрый старт

### Инициализация проекта
```bash
# При первом запуске в новой директории
opencode init-project --name "MyProject" --description "Описание проекта"
# Система создаст:
# - PROJECT.md с конституцией проекта
# - SPEC/ директорию для спецификаций модулей
# - Требования в формате Markdown
```

### Стандартный цикл работы
```bash
# 1. Анализ состояния проекта
opencode analyze-state

# 2. Обновление конституции и спецификаций (по необходимости)
opencode update-spec --module "auth" --description "Добавить OAuth2"

# 3. Планирование реализации
opencode plan-implementation --feature "user-authentication"

# 4. Разбивка на задачи
opencode break-down-tasks --feature "user-authentication"

# 5. Назначение агентов
opencode assign-agents --task-id TASK-001

# 6. TDD цикл: сначала тесты
opencode create-test --task-id TASK-001 --type "unit"

# 7. Выполнение тестов (должны провалиться initialement)
opencode run-test --task-id TASK-001

# 8. Написание кода под тесты
opencode write-code --task-id TASK-001

# 9. Запуск тестов снова (должны пройти)
opencode run-test --task-id TASK-001

# 10. Ревью кода
opencode review-code --task-id TASK-001

# 11. Интеграция и финальная проверка
opencode integrate --task-id TASK-001
```

## 🔄 Паттерн выполнения задач

Каждая задача проходит через строгий цикл:
1. **Получение назначения** - агент получает задачу с четкими критериями принятия
2. **Проверка пре-условий** - автоматическая проверка готовности к выполнению
3. **Выполнение в изолированной сессии** - работа во временной ветке с clean состоянием
4. **Самотестирование** - запуск релевантных тестов и проверок
5. **Подготовка отчета** - структурированный markdown отчет с результатами
6. **Запрос подтверждения** - показ изменений и запрос на коммит/мерж
7. **Интеграция** - после подтверждения автоматически мержит в develop

## 📊 Машина состояний проекта

Проект всегда находится в одном из состояний:
- **1**: Пустой проект (директория создана, нет ничего)
- **2**: Проект инициализирован (есть .git)
- **3**: Конституция создана (основные требования и ограничения)
- **4**: Спецификации модулей созданы (детальное описание компонентов)
- **5**: План реализации готов (оценка усилий, зависимости, риски)
- **6**: Задачи разбиты и агенты назначены (готов к выполнению)
- **7**: Тестовая фаза (написание и выполнение тестов)
- **8**: Кодинговая фаза (написание кода под тесты)
- **9**: Фаза интеграции (объединение компонентов и системное тестирование)
- **10**: Релиз-готов (все качественные gate пройдены, готово к релизу)

Переходы между состояниями блокируются до выполнения всех условий текущего состояния и получения подтверждения пользователя.

- [ ] Определен стек технологий

- [ ] Все спецификации прошли проверку на противоречия
- [ ] Доступны данные о доступности MCP серверов

### Перед написанием тестов

## 🏗️ Workflow детали (12 фаз)

### Фаза 1: Анализ состояния проекта
- Сбор информации о репозитории: язык, фреймворки, зависимости
- Определение текущего состояния машины состояний
- Проверка наличия критических файлов (README, лицензия, конфиги)
- **Вывод**: Текущее состояние проекта и рекомендации по переходу

### Фаза 2: Создание конституции проекта
- Формулирование видения проекта и целей
- Определение архитектурных решений и ограничений
- Выбор стека технологий и обоснование
- Установление правил вклада и конвенций
- **Артефакт**: `PROJECT.md`

### Фаза 3: Создание спецификаций модулей
- Декомпозиция системы на модули по принципу единой ответственности
- Для каждого модуля: входы, выходы, интерфейсы, зависимости
- Определение протоколов взаимодействия и форматов данных
- Выявление внешних интеграций и требований к ним
- **Артефакты**: `SPEC/<module_name>.md`

### Фаза 4: Анализ противоречий в спецификациях
- Проверка на дублирование функциональности между модулями
- Выявление конфликтующих требований или интерфейсов
- Анализ пропусков в покрытии требований
- Проверка реализуемости в рамках выбранного стека технологий
- **Артефакт**: `CONFLICTS_RESOLUTION.md`

### Фаза 5: Планирование реализации (ОБЩИЙ план)
- Определение порядка реализации модулей с учетом зависимостей
- Оценка трудоемкости в человеко-часах или story points
- Выявление технических рисков и стратегий их mitigation
- Планирование необходимых MCP серверов и внешних зависимостей
- **Артефакт**: `IMPLEMENTATION_PLAN.md`

### Фаза 6: Разбивка на задачи с зависимостями
- Декомпозиция модулей на атомарные задачи (максимум 8 часов работы)
- Определение четких критериев принятия для каждой задачи
- Установление зависимостей между задачами (blocked by, relates to)
- Распределение задач по типам агентов (кодер, тестировщик, рецензент)
- **Артефакт**: `TASKS.md` с таблицей задач

### Фаза 7: Анализ пробелов в требованиях
- Сравнение текущих спецификаций с оригинальными требованиями
- Выявление неочевидных сценариев использования
- Проверка соответствия нефункциональным требованиям (производительность, безопасность)
- Формулирование уточняющих вопросов для стейкхолдеров
- **Артефакт**: `REQUIREMENTS_GAP_ANALYSIS.md`

### Фаза 8: Назначение агентов на задачи
- Соответствие типа задачи специализации агента
- Учет текущей загрузки и доступности агентов
- Проверка наличия необходимых навыков и инструментов
- Назначение ответственного рецензента для каждой кодерской задачи
- **Артефакт**: `ASSIGNMENTS.md`

### Фаза 9: Создание TEST/CODE разделения (TDD First!)
- Для каждой кодерской задачи обязательно создается парная тестовая задача
- Тестовые задачи получают префикс `TEST:` и выполняются первыми
- Определение уровня юнит-тестирования vs интеграционного тестирования
- Установление минимального порога покрытия тестами (по умолчанию 80%)
- **Артефакт**: Обновление `TASKS.md` с пометкой TEST задач

### Фаза 10: TDD Координация (языковая специализация)
- Назначение тестовых агентов, специализированных на языке проекта
- Обеспечение доступности необходимых тестовых фреймворков и библиотек
- Настройка изолированных сред для выполнения тестов (контейнеры, виртуальные окружения)
- Установление правил написания тестовых данных и моков
- **Артефакт**: `TEST_ENVIRONMENT_SETUP.md`

### Фаза 11: Разработка кода под тесты
- Написание минимума кода для прохождения конкретного теста
- Рефакторинг с сохранением прохождения всех тестов
- Следование принципам чистого кода и соглашениям проекта
- Ежечасный коммит с описательными сообщениями
- **Артефакты**: Исходный код в соответствующих директориях

### Фаза 12: Выполнение задач и интеграция
- Запуск полного тест набора (юнит, интеграционный, системный)
- Проверка соответствия нефункциональным требованиям
- Подготовка артефактов релиза (сборка, документация, заметки о релизе)
- Финальное рецензирование интегрированного решения
- **Артефакт**: Готовый к релизу кандидат

## 🛡️ Quality Gates детали

### Gate 1: Pre-Execution (Синтаксическая валидность)
- Проверка: код компилируется без ошибок
- Инструменты: `tsc --noEmit` для TypeScript, `python -m py_compile` для Python
- Порог: 100% файлов должны проходить

### Gate 2: Post-Execution (Тестовое покрытие)
- Проверка: покрытие unit тестами ≥ 80%
- Инструменты: `jest --coverage`, `pytest --cov`, или эквивалент
- Порог: среднее по проекту ≥ 80%, никакой файл < 60%

### Gate 3: Pre-Commit (Безопасность linting)
- Проверка: отсутствие критических уязвимостей
- Инструменты: `npm audit`, `bandit`, `eslint security плагины`
- Порог: нулевые критические предупреждения

### Gate 4: Pre-Merge (Интеграционные проверки)
- Проверка: ветка влита в develop, нет конфликтов
- Инструменты: `git branch --merged`, `git status --porcelain`
- Порог: нет конфликтов, требуется подтверждение пользователя

### Gate 5: Pre-Implementation (Соответствие спецификациям)
- Проверка: реализованы все требования из спецификаций модулей
- Инструменты: трассировка требований к тестам (RTM)
- Порог: 100% покрытия требований тестами с проходом

### Gate 6: MCP Check (Доступность MCP серверов)
- Проверка: доступны необходимые MCP серверы для технологий проекта
- Инструменты: `mcpResolution.detectTechnologies()`, `mcpResolution.searchMCPServers()`
- Порог: критические MCP (filesystem, git, memory) доступны

## ⚙️ Speckit workflow детали

### TDD First! enforce
- Тестовые задачи всегда создаются перед кодовыми
- Переход к кодовой задаче блокируется до прохождения тестовых
- Тестовые задачи имеют высший приоритет в планировании

### Изолированные сессии
- Каждая задача выполняется в отдельной ветке feature/
- Рабочая директория чистая от локальных изменений
- После завершения автоматически возвращается к develop для сбора изменений

### Языково-специализированные агенты
- Тестовые агенты подбираются под язык проекта (pytest для Python, Jest для TS/JS)
- Кодерские агенты имеют опыт работы со стеком технологий проекта
- Аналитики работают с доменной спецификой и методами исследования

### Автоматическая регрессия
- После каждого успешного мержа запускается полный тест набор
- Любое падение теста блокирует дальнейшую работу до исправления
- Ведется журнал регрессий с анализом причин

## 🔌 MCP Dynamic Resolution System

### Принцип работы
1. При начале фазы реализации система анализирует:
   - Требуемые технологии из спецификаций модулей
   - Текущий стек проекта из конституции
   - Доступность локальных инструментов и SDK
2. Для недостающих компонентов выполняется поиск:
   - В локальном реестре MCP серверов
   - В публичных репозиториях (npm, PyPI, crates.io и т.д.)
   - По известным альтернативам и заглушкам
3. Выбранные MCP серверы автоматически:
   - Скачиваются и устанавливаются во временную директорию
   - Конфигурируются под текущий проект
   - Регистрируются в системе для использования агентами
4. Кеширование успешно настроенных серверов для повторного использования
5. Автоматическое обновление при обнаружении новых версий (с подтверждением пользователя)

### Интеграция с workflow
- Проверка доступности MCP выполняется в Gate перед фазой 11
- Кеш хранится в `.opencode/mcp-cache/` с версионированием
- Система журнал всех операций установки и конфигурации
- Откат к предыдущей версии при проблемах с новым сервером

## 💾 Сохранение контекста

### Механизм
- После каждой успешной операции сохраняется снимок состояния:
  - Текущее состояние машины состояний
  - Список выполненных задач и их результаты
  - Конфигурация активных MCP серверов
  - Текущие настройки и переменные окружения
- Сохранение в `.opencode/context/` в формате JSON с timestamp
- Максимум 10 последних контекстов хранится локально
- При перезагрузке системы предлагается восстановить последний известный хороший контекст

### Использование
- Восстановление после прерывания работы (кранш, перезагрузка)
- Сравнение текущего состояния с эталонным для обнаружения дрейфа
- Откат к предыдущему состоянию при обнаружении критических ошибок
- База для генерации отчетов о прогрессе проекта

## 📚 Приложения

### Пример типичного диалога агента
```
Агент: Задача TASK-001 (написать unit тест для функции login) прошла pre-condition проверку.
        Текущее состояние проекта: 40 (задачи разбиты)
        Подготовлено к написанию теста в src/auth/login.test.ts

        Отчет будет включать:
        - Создан/обновлен тестовый файл
        - Результат выполнения теста (должен провалиться inicialmente)
        - Следующий шаг: написание кода для прохождения теста

        Продолжить? (да/нет)
```

### Типичные ошибки и способы их решения
| Ошибка | Причина | Решение |
|--------|---------|---------|
| Aгент пытается писать код без тестов | Нарушение TDD First! правила | Блокировать операцию, направить на написание теста |
| Попытка мерж незавершенной feature ветки | Пропуск предварительных проверок | Обязать запустить тесты и получить подтверждение |
| Использование неподходящего MCP сервера | Неправильная идентификация технологии | Добавить правила маппинга технологий в конфигурацию |
| Потеря контекста между сессиями | Не сохранено состояние перед перерывом | Включить автоматическое сохранение после каждой операции |
| Конфликт версий зависимостей | Обновление MCP без тестирования | Ввести период тестирования для обновленных серверов |

### Полезные команды и алиасы
```bash
# Быстрый анализ состояния
opencode state

# Показ текущих задач
opencode tasks --status in_progress

# Запуск полной проверки качества
opencode quality-gate --all

# Просмотр последних отчетов
opencode reports --limit 5

# Очистка кэша и временных файлов (с подтверждением)
opencode cleanup --confirm
```

## 🔄 TDD Workflow (TEST/CODE)

### Обязательное разделение
Каждая кодовая задача требует先 TEST задачи:

```
1. TEST: Написать тесты (должны упасть - RED)
2. CODE: Написать код (тесты проходят - GREEN)
3. REFACTOR: Рефакторинг
```

### Проверки
- `tddWorkflow.registerTestTask()` — связать TEST и CODE задачи
- `tddWorkflow.canStartCode()` — проверить готовность
- `tddWorkflow.tryStartCodeTask()` — начать кодовую задачу

### Блокировка
Кодер **не может** начать работу, пока тесты не пройдены.
