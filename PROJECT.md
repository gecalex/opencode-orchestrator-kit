# Конституция проекта Orchestrator Kit

Документ определяет фундаментальные принципы, правила и ограничения проекта Orchestrator Kit.

---

## 1. Видение проекта

### 1.1 Общее описание

**Orchestrator Kit** — это плагин для opencode, реализующий TDD-ориентированную методологию разработки с явным подтверждением пользователя на каждом этапе workflow.

Плагин автоматизирует процесс разработки программного обеспечения, обеспечивая строгое следование методологии Test-Driven Development и гарантируя качество кода через систему Quality Gates.

### 1.2 Позиционирование

| Характеристика | Значение |
|----------------|-----------|
| Тип | Плагин (extension) для opencode |
| Язык реализации | TypeScript |
| Целевая платформа | opencode CLI |
| Версия Node.js | >= 18.0.0 |

### 1.3 Ключевые ценности

1. **Предсказуемость** — каждый этап workflow определён и понятен
2. **Контроль** — пользователь всегда подтверждает критические решения
3. **Качество** — встроенные проверки качества на каждом этапе
4. **Прозрачность** — полная видимость состояния проекта в любой момент

---

## 2. Цели проекта

### 2.1 Бизнес-цели

- Автоматизация workflow разработки от инициализации до релиза
- Обеспечение строгого TDD-процесса с обязательным разделением TEST/CODE фаз
- Интеграция системы Quality Gates для автоматического контроля качества
- Поддержка динамического разрешения MCP серверов по требованиям задачи

### 2.2 Технические цели

| Цель | Описание | Метрика |
|------|----------|---------|
| State Machine | 10 состояний проекта от 0 до 10 | Полное покрытие |
| Quality Gates | 6 ворот качества | Прохождение 100% |
| Agent System | 16 специализированных агентов | Работоспособность |
| MCP Registry | 20+ MCP серверов | Доступность |

### 2.3 Методологические цели

- **TDD First!** — тесты пишутся до кода, код пишется под тесты
- **Изолированные сессии** — каждая задача в отдельной feature-ветке
- **Языковая специализация** — агенты выбираются под язык проекта
- **Автоматическая регрессия** — полный тест-набор после каждого мержа

### 3.2 Архитектурные принципы

#### Принцип 1: Plugin-Based Architecture

Плагин загружается как расширение opencode и предоставляет:
- Кастомные инструменты (`speckit-*`)
- Агентов для различных задач
- Скиллы для специализированных операций

#### Принцип 2: State Machine

Проект всегда находится в одном из 10 состояний:

| Код | Состояние | Описание |
|-----|----------|----------|
| 0 | Пустой проект | Директория создана, нет ничего |
| 1 | Инициализирован | Есть .git |
| 2 | Конституция создана | PROJECT.md готов |
| 3 | Спецификации готовы | SPEC/*.md созданы |
| 4 | План реализации | IMPLEMENTATION_PLAN.md |
| 5 | Задачи разбиты | TASKS.md с назначениями |
| 6 | Тестовая фаза | TEST:* задачи в работе |
| 7 | Кодинговая фаза | CODE:* задачи в работе |
| 8 | Фаза интеграции | Компоненты объединяются |
| 9 | Релиз-готов | Все gates пройдены |

**Переходы** — каждое состояние блокирует переход к следующему до выполнения всех условий текущего.

#### Принцип 3: Agent System

16 специализированных агентов:

| Агент | Назначение | Состояния |
|-------|-----------|-----------|
| project-initializer | Инициализация проекта | 0 → 1 |
| constitution-agent | Создание конституции | 1 → 2 |
| specify-agent | Создание спецификаций | 2 → 3 |
| plan-agent | Планирование реализации | 3 → 4 |
| task-agent | Разбивка на задачи | 4 → 5 |
| python-developer | Python разработка | 6 → 7 → 8 |
| typescript-developer | TypeScript разработка | 6 → 7 → 8 |
| test-developer | Написание тестов | 5 → 6 |
| code-reviewer | Рецензирование кода | 7 → 8 |
| security-auditor | Аудит безопасности | 8 |
| integration-tester | Интеграционное тестирование | 8 |
| mcp-manager | Управление MCP серверами | Любое |
| documentation-writer | Документация | Любое |
| git-agent | Git операции | Любое |
| meta-agent | Анализ и оптимизация | Любое |

#### Принцип 4: MCP Dynamic Resolution

Система динамического разрешения MCP серверов:

1. **Анализ требований** — определение технологий проекта из спецификаций
2. **Поиск серверов** — обращение к реестру MCP (20+ серверов в 12 категориях)
3. **Установка** — автоматическая установка недостающих серверов
4. **Кеширование** — сохранение конфигурации для повторного использования

## 4. Ограничения

### 4.1 Критические ограничения

#### Ограничение 1: Явное подтверждение пользователя

**ПРАВИЛО:** На каждом критическом переходе требуется явное подтверждение пользователя.

Критические переходы:
- Инициализация → после проверки пре-условий
- Анализ → Планирование (фаза 3 → 4)
- Планирование → Выполнение (фаза 6 → 7)
- TEST → CODE (после написания тестов)
- CODE → Интеграция (после написания кода)
- Любое изменение архитектуры или базы данных

**ФОРМАТ подтверждения:**
```
✅ Фаза [N] завершена!

Результат:
- [результат 1]
- [результат 2]

Следующий шаг: [название]

Продолжить? (да/нет)
```

#### Ограничение 2: TDD First!

**ПРАВИЛО:** Тестовые задачи (TEST:*) создаются и выполняются ПЕРЕД кодовыми задачами (CODE:*).

- Кодер **не может** начать работу, пока тесты не провалились (RED)
- Переход от TEST к CODE блокируется до прохождения тестов
- Минимальное покрытие тестами: 80%

#### Ограничение 3: Quality Gates

**ПРАВИЛО:** Каждый этап должен пройти соответствующие Quality Gates.

| Gate | Проверка | Порог |
|------|----------|-------|
| 1: Pre-Execution | Компиляция | 100% файлов |
| 2: Post-Execution | Покрытие тестами | ≥ 80% |
| 3: Pre-Commit | Безопасность | 0 критических |
| 4: Pre-Merge | Интеграция | Нет конфликтов |
| 5: Pre-Implementation | Соответствие спецификациям | 100% |
| 6: MCP Check | Доступность MCP | Все доступны |

#### Ограничение 4: Изолированные сессии

**ПРАВИЛО:** Каждая задача выполняется �� отдельной feature-ветке.

- Рабочая директория чистая от локальных изменений
- После завершения — автоматический возврат к develop
- Коммит только после подтверждения пользователя

### 4.2 Технические ограничения

| Ограничение | Описание |
|-------------|----------|
| Node.js | >= 18.0.0 |
| TypeScript | >= 5.0.0 |

### 4.3 Блокирующие правила

#### Аварийный тормоз

Если пре-условия не выполнены:
1. Останавливаем процесс
2. Возвращаем управление пользователю
3. Предлагаем выполнить необходимое действие

#### Проверка пре-условий

Перед запуском любого агента проверяется:

- [ ] `.gitignore` существует
- [ ] Pre-Flight проверки пройдены
- [ ] Пользователь подтвердил переход

## 5. Правила вклада

### 5.1 QWEN Philosophy

Проект следует философии QWEN (из QWEN Orchestrator Kit):

1. **Безопасность превыше скорости** — никогда не пропускать проверки ради скорости
2. **Подтверждение — обязательно** — никаких автоматических переходов без согласия
3. **Качество — не обсуждается** — 100% соответствие Quality Gates
4. **Прозрачность — по умолчанию** — пользователь видит всё

### 5.2 Процесс разработки

#### Стандартный цикл

```
1. analyze-state    → Определение текущего состояния
2. init-project   → Инициализация (только для новых проектов)
3. create-constitution → Создание PROJECT.md
4. create-specifications → SPEC/*.md для модулей
5. plan-implementation → IMPLEMENTATION_PLAN.md
6. break-down-tasks → TASKS.md с задачами
7. create-test → TEST:* задачи (TDD First!)
8. write-code → CODE:* задачи
9. review-code → Проверка качества
10. integrate → Merge в develop
```

#### Git Workflow

1. Все изменения в feature-ветках (`feature/*`, `fix/*`, `hotfix/*`)
2. Merge только через Pull Request
3. Squash при желании
4. Conventional Commits обязательны

### 5.3 Стиль кода

| Правило | Стандарт |
|---------|----------|
| TypeScript | strict mode |
| ESLint | eslint-plugin-security |
| Форматирование | prettier |
| Тесты | Jest |
| Покрытие | >= 80% |

### 5.4 Документация

- Все ответы — на русском языке
- Все артефакты — в формате Markdown
- Коммиты — Conventional Commits
- Отчёты — в REPORTS/*

## 7. Quality Gates

### 7.1 Определение ворот

#### Gate 1: Pre-Execution

- **Цель:** Проверка синтаксической валидности
- **Инструменты:** `tsc --noEmit`, `python -m py_compile`
- **Порог:** 100% файлов компилируются без ошибок

#### Gate 2: Post-Execution

- **Цель:** Проверка тестового покрытия
- **Инструменты:** `jest --coverage`, `pytest --cov`
- **Порог:** ≥ 80% среднее по проекту, ≥ 60% минимум для файла

#### Gate 3: Pre-Commit

- **Цель:** Проверка безопасности кода
- **Инструменты:** `npm audit`, `bandit`, `eslint security`
- **Порог:** 0 критических уязвимостей

#### Gate 4: Pre-Merge

- **Цель:** Проверка интеграции
- **Инструменты:** `git status`, `git diff`
- **Порог:** Нет конфликтов, подтверждение пользователя

#### Gate 5: Pre-Implementation

- **Цель:** Проверка соответствия спецификациям
- **Инструменты:** Тр��ссировка требований к тестам (RTM)
- **Порог:** 100% требований покрыты тестами

#### Gate 6: MCP Check

- **Цель:** Проверка доступности MCP серверов
- **Инструменты:** `mcpResolution.detectTechnologies()`
- **Порог:** Критические MCP доступны (filesystem, git, memory)

### 7.2 Прохождение ворот

Перед каждым мержа в develop:
1. Пройти Gate 1 (компиляция)
2. Пройти Gate 2 (покрытие)
3. Пройти Gate 3 (безопасность)
4. Пройти Gate 4 (merge)
5. Подтверждение пользователя

## 9. Версионирование

### 9.1 Схема версий

Используется SemVer:

```
MAJOR.MINOR.PATCH
```

- **MAJOR:** Несовместимые изменения API
- **MINOR:** Новая функциональность (обратная совместимость)
- **PATCH:** Исправления багов (обратная совместимость)

### 9.2 Тегирование

Теги в формате: `v{version}`

Пример: `v1.0.0`
