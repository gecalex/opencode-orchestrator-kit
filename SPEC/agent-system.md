# Спецификация модуля agent-system

## 1. Обзор

Модуль agent-system описывает систему 16 специализированных агентов Orchestrator Kit. Каждый агент имеет четко определенную зону ответственности, набор инструментов и разрешенные состояния проекта, в которых он может выполнять задачи.

## 2. Функциональность

- Определение 16 специализированных агентов с их назначениями
- Управление назначением агентов на задачи в зависимости от типа задачи и состояния проекта
- Обеспечение изоляции 책임ственности между агентами (каждый агент отвечает за свою область)
- Координация работы агентов через оркестратор
- Предоставление информации о доступности и загруженности агентов
- Управление жизненным циклом агентов (инициализация, выполнение, завершение)

## 3. Агенты и их назначения

| Агент | Назначение | Состояния проекта | Ключевые ответственности |
|-------|------------|-------------------|--------------------------|
| project-initializer | Инициализация проекта | 0 → 1 | Проверка расширения, инициализация Git, создание .gitignore |
| constitution-agent | Создание конституции | 1 → 2 | Формулирование видения проекта, определение архитектурных решений, выбор стека технологий |
| specify-agent | Создание спецификаций | 2 → 3 | Создание спецификаций модулей, требований, глоссария, краткого содержания |
| plan-agent | Планирование реализации | 3 → 4 | Определение порядка реализации модулей, оценка трудоемкости, выявление рисков |
| task-agent | Разбивка на задачи | 4 → 5 | Декомпозиция модулей на атомарные задачи, определение критериев принятия, установление зависимостей |
| python-developer | Python разработка | 6 → 7 → 8 | Написание тестов (TEST:*), написание кода под тесты (CODE:*), рефакторинг |
| typescript-developer | TypeScript разработка | 6 → 7 → 8 | Написание тестов (TEST:*), написание кода под тесты (CODE:*), рефакторинг |
| test-developer | Написание тестов | 5 → 6 | Создание тестовых сценариев, написание unit и интеграционных тестов, настройка тестовой среды |
| code-reviewer | Рецензирование кода | 7 → 8 | Проверка качества кода, обеспечение соответствия стандартам, выявление потенциальных проблем |
| security-auditor | Аудит безопасности | 8 | Проверка уязвимостей, поиск secrets, анализ соответствия OWASP Top 10 |

| meta-agent | Анализ и оптимизация | Любое | Анализ производительности, оптимизация workflow, выявление узких мест, предложения улучшений |

## 4. Интерфейсы

### Входы:
- Назначение задачи от оркестратора (taskId, тип задачи, описание, критерии принятия)
- Текущее состояние проекта (от state-machine.ts)
- Подтверждение пользователя на критических переходах
- Результаты выполнения предшествующих задач
- Конфигурация и параметры для выполнения задачи

### Выходы:
- Результат выполнения задачи (успех/неудача)
- Обновленное состояние проекта (если применимо)
- Созданные/измененные файлы и артефакты
- Отчет о выполненной работе (в формате Markdown)
- Обновления в системе задач (TASKS.md, ASSIGNMENTS.md и т.д.)
- Уведомления о необходимости действий пользователя

### Внутренние методы:
- `executeTask(task: Task): Promise<TaskResult>` - основной метод выполнения задачи
- `validatePreConditions(): Promise<PreConditionResult>` - проверка пре-условий перед выполнением
- `postExecute(): Promise<void>` - действия после выполнения задачи (сохранение результатов, уведомления)
- `getCapabilities(): AgentCapabilities` - получение информации о возможностях агента
- `handleUserApproval(decision: boolean): Promise<void>` - обработка решения пользователя

## 5. Ограничения

- Агенты могут выполнять задачи только в разрешенных состояниях проекта
- Каждый агент имеет строго определенную зону ответственности и не должен выходить за ее пределы
- Переход к кодовой задаче (CODE:*) блокируется до прохождения соответствующих тестовых задац (TEST:*)
- Агенты должны следовать принципу TDD First! где применимо
- Все операции агентов должны проходить соответствующие Quality Gates
- Агенты должны работать в изолированных feature-ветках
- Агенты должны получать явное подтверждение пользователя на критических переходах
- Агенты должны документировать свою работу в структурированном формате Markdown

## 6. Внешние интеграции

- Взаимодействует с оркестратором (orchestrator.ts) для получения задач и отправки результатов
- Интегрируется с state-machine.ts для проверки разрешенности выполнения в текущем состоянии
- Взаимодействует с blocking-rules.ts для проверки пре-условий перед выполнением задач
- Использует permissions.ts для проверки разрешений на выполнение операций
- Взаимодействует с workflow-модулями (pre-flight, tdd-workflow, git-workflow, quality-gates) для выполнения специализированных операций
- Интегрируется с MCP-системой (mcp-registry.ts, mcp-resolution.ts) для доступа к внешним серверам
- Взаимодействует с системой сессий (session-hooks.ts, review-sessions.ts) для обработки событий сессии
- Использует reporting.ts для генерации отчетов о выполненной работе
- Взаимодействует с файловой системой для чтения/записи файлов проекта

## 7. Требования

1. Должен предоставлять четкое определение 16 специализированных агентов с их зонами ответственности
2. Каждый агент должен иметь определенные разрешенные состояния проекта для выполнения задач
3. Должен обеспечивать механизм назначения агентов на задачи в зависимости от типа задачи и состояния проекта
4. Должен обеспечивать изоляцию ответственности между агентами
5. Должен поддерживать принцип TDD First! для разработческих агентов
6. Должен обеспечивать проверку пре-условий перед выполнением задач
7. Должен интегрироваться с оркестратором для получения задач и отправки результатов
8. Должен поддерживать работу в изолированных feature-ветках
9. Должен обеспечивать явное подтверждение пользователя на критических переходах
10. Должен генерировать структурированные отчеты о выполненной работе
11. Должен взаимодействовать с системой Quality Gates для проверки качества выполненной работы
12. Должен поддерживать MCP-интеграцию для доступа к внешним серверам при необходимости
