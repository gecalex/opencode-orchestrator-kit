# Спецификация модуля session-management

## 1. Обзор

Модуль session-management отвечает за обработку событий сессии и управление ревью сессий в Orchestrator Kit. Он состоит из двух компонентов: session-hooks.ts (обработка событий сессии) и review-sessions.ts (управление ревью сессий).

## 2. Функциональность

### 2.1 Session Hooks (session-hooks.ts)
- Обработка событий сессии Orchestrator Kit:
  - `session.created` - начало новой сессии
  - `session.idle` - завершение сессии без активности
  - `session.compacted` - компактация контекста сессии
  - `session.error` - возникновение ошибки в сессии
- Сохранение и загрузка контекста сессии между операциями
- Интеграция с state-machine.ts для отслеживания изменений состояния проекта
- Предоставление данных для восстановления состояния после перезагрузки
- Журналирование событий сессии для аудита и отладки

### 2.2 Review Sessions (review-sessions.ts)
- Управление процессом код ревью через GitHub интеграцию
- Создание и управление pull request'ами для ревью кода
- Сбор и агрегация комментариев к pull request'ам
- Отслеживание статуса проверок и статусов pull request'ов
- Facilitation процесса одобрения изменений через conventional reviews
- Интеграция с git-workflow.ts для автоматического мерж после одобрения
- Управление lifecycle pull request'ов (создание, обновление, закрытие, мерж)

## 3. Интерфейсы

### Входы для Session Hooks:
- События сессии от opencode host
- Запросы на сохранение контекста сессии
- Запросы на загрузку контекста сессии
- Изменения состояния проекта (от state-machine.ts)
- Ошибки и исключения во время выполнения операций

### Выходы из Session Hooks:
- Сохраненный контекст сессии в хранилище (.opencode/context/)
- Загруженный контекст сессии для восстановления состояния
- Уведомления о событиях сессии для оркестратора
- Статус готовности сессии к выполнению операций
- Журнал событий сессии для аудита

### Входы для Review Sessions:
- Запросы на создание pull request'а для ревью кода
- Информация о выполненных изменениях (файлы, коммиты)
- Данные о текущей ветке и целевой ветке для мерж
- Результаты проверок качества и тестов
- Решения пользователя о необходимости изменений
- Комментарии и ревью от reviewers

### Выходы из Review Sessions:
- Созданные pull request'ы в GitHub репозитории
- Обновленные pull request'ы с дополнительной информацией
- Статус pull request'ов (открыт, закрыт, merged)
- Собранные комментарии и ревью к pull request'ам
- Уведомления о необходимости действий (обработать комментарии, внести исправления)
- Подтверждение готовности к мерж pull request'а

### Взаимодействие между компонентами:
- Session Hooks сохраняет контекст перед созданием review session
- Review Sessions использует сохраненный контекст для восстановления состояния после ревью
- Оба компонента взаимодействуют с оркестратором для получения команд и отправки результатов
- Review Sessions интегрируется с git-workflow.ts для операций с ветками и мержем

## 4. Ограничения

- Контекст сессии должен сохраняться после каждой успешной операции
- Контекст сессии должен включать: текущее состояние проекта, список выполненных задач, конфигурацию MCP серверов, настройки и переменные окружения
- Должно сохраняться не более 10 последних контекстов сессии для предотвращения переполнения хранилища
- При перезагрузке системы должно предлагаться восстановление последнего известного хорошего контекста
- Review Sessions должен поддерживать только авторизованные операции с GitHub репозиторием
- Review Sessions должен требовать явного подтверждения пользователя перед созданием pull request'а
- Review Sessions должен блокировать мерж pull request'а до прохождения всех проверок и получения одобрения
- Session Hooks должен обеспечивать потокобезопасность при concurrent доступе к контексту сессии
- Review Sessions должен обрабатывать ошибки сети и временные недоступности GitHub API

## 5. Внешние интеграции

- Взаимодействует с оркестратором (orchestrator.ts) для получения команд и отправки результатов
- Интегрируется с state-machine.ts для отслеживания и сохранения состояния проекта
- Используется модулем quality-gates.ts для передачи информации о прохождении ворот
- Взаимодействует с agent-system для контекстной информации о выполняемых задачах
- Интегрируется с workflow-modules (pre-flight, tdd-workflow, git-workflow) для координации операций
- Взаимодействует с mcp-integration для сохранения и восстановления конфигурации MCP серверов
- Review Sessions интегрируется с GitHub API через octokit или аналогичную библиотеку
- Review Sessions взаимодействует с git-workflow.ts для git операций (создание веток, коммиты, мерж)
- Оба компонента используют файловые операции для работы с локальным хранилищем контекста (.opencode/context/)
- Оба компонента взаимодействуют с системой разрешений (permissions.ts) для проверки прав на операции

## 6. Требования

1. Session Hooks должен обрабатывать все четыре типа событий сессии: создан, простаивает, компактирован, ошибка
2. Session Hooks должен сохранять контекст сессии после каждой успешной операции в формате JSON с timestamp
3. Session Hooks должен загружать контекст сессии при восстановлении после перезагрузки или продолжении работы
4. Session Hooks должен ограничивать количество сохраненных контекстов (максимум 10) с удалением самых старых
5. Session Hooks должен предоставлять механизм восстановления последнего известного хорошего контекста при ошибках
6. Review Sessions должен поддерживать создание pull request'ов для ревью кода с автоматическим заполнением шаблона
7. Review Sessions должен поддерживать обновление существующих pull request'ов новыми коммитами и информацией
8. Review Sessions должен отслеживать статус проверок (CI/CD) и статусов pull request'ов
9. Review Sessions должен собирать и агрегировать комментарии к pull request'ам для представления пользователю
10. Review Sessions должен обеспечивать процесс одобрения изменений через conventional reviews (APPROVE, REQUEST_CHANGES, COMMENT)
11. Review Sessions должен интегрироваться с git-workflow.ts для автоматического мерж pull request'а после одобрения и прохождения всех проверок
12. Review Sessions должен требовать явного подтверждения пользователя на критических этапах (создание PR, мерж PR)
13. Оба компонента должны обрабатывать ошибки gracefully и предоставлять информативные сообщения пользователю
14. Оба компонента должны следовать принципу минимальных привилегий при работе с внешними системами
15. Оба компонента должны обеспечивать аудит всех операций через журнал событий
