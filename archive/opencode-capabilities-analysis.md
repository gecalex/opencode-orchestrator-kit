# Анализ возможностей реализации системы оркестрации на базе OpenCode

## Выводы исследования

### Текущие возможности OpenCode для реализации оркестрации

#### 1. Архитектура плагинов (официальная документация)
- **Контекст плагина**: получает `project`, `client`, `$`, `directory`, `worktree`
- **Поддерживаемые события**: 
  - Сессия: `session.created`, `session.idle`, `session.compacted`, `session.deleted`, `session.error`
  - Инструменты: `tool.execute.before`, `tool.execute.after`
  - Файлы: `file.edited`, `file.watcher.updated`
  - Todo: `todo.updated`
  - Команды: `command.executed`
  - Shell: `shell.env`
- **Возможности**:
  - Регистрация кастомных инструментов через `tool` helper
  - Взаимодействие с AI через `client.session.prompt()`
  - Выполнение команд через `$.command`
  - Чтение/запись файлов через стандартные инструменты
  - Поддержка TypeScript
  - Порядок загрузки: глобальный конфиг → проектный конфиг

#### 2. Доказуемо реализованные подходы в существующих плагинах

**A. Подход на основе конечного автомата (FSM)**
- Используется в: Open Artisan, Micode
- Особенности:
  - Табличный детерминированный конечный автомат
  - Фазы workflow с явными переходами
  - Каждая фаза следует циклу: DRAFT → REVIEW → USER_GATE → (редактирование) → REVISE → REVIEW
  - Структурное предотвращение пропуска фаз
  - Примеры состояний: DISCOVERY, PLANNING, INTERFACES, TESTS, IMPL_PLAN, IMPLEMENTATION, DONE

**B. Многоагентная оркестрация**
- Используется в: Oh My OpenCode, OpenCode Swarm
- Особенности:
  - Специализированные агенты для разных ролей (планировщик, кoder, рецензент, тестировщик)
  - Возможность параллельного выполнения
  - Изолированные рабочие деревья (worktrees) для безопасной экспериментации
  - Агрегация результатов и обнаружение конфликтов
  - Примеры агентов: архитектор, критик, кодер, тестировщик, документатор

**C. Принудительное выполнение workflow**
- Используется в: Open Artisan, Micode
- Особенности:
  - Обязательные фазы, которые нельзя пропустить
  - Контрольные точки качества на каждом переходе
  - Контрольные точки утверждения пользователя
  - Изолированные сессии рецензирования (эфемерные сабагенты для валидации)
  - Примеры: проверка интерфейсов перед реализацией, TDD-подход (тесты перед кодом)

**D. Механизмы persistence**
- Используется в: практически всех плагинах
- Особенности:
  - Хранение состояния сессии в JSON файлах
  - Изолированные рабочие деревья для реализации
  - Сохранение контекста между сессиями
  - Системы checkpointing
  - Примеры: `.opencode/full-auto/state.json`, `.swarm/` директория

**E. Регистрация кастомных инструментов** (из предложения в GitHub issue #20849)
- Потенциальные инструменты для оркестрации:
  - `background` tool (fire-and-forget агенты): создает дочерние сессии через SDK, немедленно возвращает управление
  - `team` tool (fan-out с DAG-зависимостями): запускает параллельные агентные сессии с управлением зависимостями
  - `team_status` tool (мониторинг): отслеживание запущенных агентов
  - Потенциал для месагентного общения: `team_send` / `team_recv` инструменты

### Оценка осуществимости реализации функций Qwen Orchestrator Kit в OpenCode

#### ✅ Высокая осуществимость (уже доказано в существующих плагинах):
1. **Базовая машина состояний** - реализована в Open Artisan, Micode
2. **Pre-Flight проверки** - возможна через хуки событий `session.created`
3. **Базовое разделение агентов** - реализовано в Oh My OpenCode, Swarm
4. **Инструментальные ограничения** - возможна через `tool.execute.before` хук
5. **Базовая система промптов** - стандартная функция плагинов
6. **Изолированное выполнение** - через worktree и эфемерные сессии
7. **Пользовательские утверждения** - реализовано в Open Artisan как USER_GATE
8. **Сессионная персистентность** - стандартная практика в плагинах

#### ⚠️ Средняя осуществимость (требует разработки, но выполнимо):
1. **MCP Dynamic Resolution System** - требует интеграции с MCP клиент/сервер, но возможно через кастомные инструменты
2. **Метапрограммирование агентов** - создание агентов динамически сложно, но возможно через запись файлов и перезагрузку конфигурации
3. **Детализированная 12-фазовая система workflow** - возможна через расширенный FSM, аналогично Open Artisan
4. **Блокирующие правила с автоматическими проверками** - возможна через предварительные хуки инструментов
5. **Детальная система отчетности** - можно реализовать через кастомные инструменты и форматированный вывод
6. **Принцип минимальных привилегий** - возможен через строгую матрицу разрешений в хуках

#### ❌ Низкая осуществимость (требует изменений ядра или имеет серьезные ограничения):
1. **Полная изоляция состояния** - состояние в основном хранится в памяти плагина
2. **Автоматическая перезагрузка агентов** - требует перезапуска сессий или сложных механизмов
3. **Глубокая интеграция с ядром OpenCode** - некоторые аспекты могут потребовать изменений upstream
4. **Полная блокировка определённых действий на ядерном уровне** - ограничено возможностями хуков

### Рекомендуемый формат реализации для максимальной полноты

На основе исследования существующих подходов и возможностей OpenCode, рекомендуется следующий формат:

#### 1. **Техническая стек**
- **Язык**: TypeScript (для лучшей типизации и поддержки IDE)
- **Структура**: Стандартная структура плагина OpenCode с расширением для оркестрации
- **Сборка**: Bun/npm для управления зависимостями и сборки

#### 2. **Компоненты архитектуры**

**A. Ядро оркестрации (Табличный FSM)**
- Расширенная машина состояний с 12+ фазами, соответствующими Speckit workflow
- Каждая фаза имеет явные условия входа/выхода
- Обязательные контрольные точки с проверкой пре-условий
- Автоматический возврат управления оркестратору после каждого сабагента

**B. Система кастомных инструментов**
- `speckit-analyze-state` - определение текущего состояния проекта
- `speckit-mcp-resolve` - динамическое разрешение и настройка MCP серверов
- `speckit-agent-spawn` - создание специализированных агентов на лету
- `speckit-background` - fire-and-forget выполнение задач
- `speckit-team` - координация параллельных агентных сессий с DAG-зависимостями
- `speckit-report-generator` - создание структурированных отчетов о делегировании

**C. Система хуков и событий**
- `session.created` - запуск Pre-Flight проверок и определение состояния
- `session.idle` - сохранение контекста и предложение следующих шагов
- `tool.execute.before` - предварительная валидация инструментов и проверка разрешений
- `tool.execute.after` - пост-исполнительная верификация и обновление состояния

**D. Система изолированного рецензирования**
- Эфемерные сабагенты для проверки артефактов на каждом этапе
- Отдельные сессии для рецензирования без доступа к коду автора
- Критерии приемлемости, основанные на спецификациях и конституции

**E. Механизм TDD First!**
- Обязательное разделение TEST/CODE задач
- Изолированные сессии для написания тестов перед реализацией
- Проверка покрытия и качества тестов перед переходом к коду
- Автоматическое определение языка и подбор соответствующих тестовых фреймворков

**F. Система сохранения контекста**
- Персистентное хранение состояния workflow в JSON
- Сессии компактации с сохранением важного контекста
- Восстановление сессий с сохранением прогресса
- Checkpointing для длительных операций

#### 3. **Модель безопасности и разрешений**
- Матрица разрешений для каждого типа агента (оркестратор, рецензент, кодер, тестировщик и т.д.)
- Строгое ограничение доступа к файловой системе на основе ролей
- Запрет прямого доступа к `write_file` для оркестраторов (только через инструменты задач)
- Изолированное выполнение потенциально опасных операций

#### 4. **Документация и пользовательский опыт**
- Комплексный файл `AGENTS.md` в качестве основной документации (аналогично оригиналу)
- Система подсказок и предложений следующих шагов на каждой фазе
- Обязательные диалоги подтверждения пользователя перед критическими операциями
- Структурированная система отчетов о делегировании и прогрессе

#### 5. **Интеграция с MCP**
- Динамический поиск MCP серверов на основе технологий из спецификаций
- Автоматическая установка и настройка найденных серверов
- Интеграция с workflow перед фазой реализации
- Кеширование и повторное использование настроек MCP

### Оценка реалистичности достижения полной функциональности

На основе анализа существующих плагинов и возможностей OpenCode:

**Достижимо на 80-90%:**
- Базовая машина состояний и workflow enforcement
- Pre-Flight проверки и базовые качественные gate
- Многоагентная оркестрация с специализированными ролями
- Изолированное выполнение и worktree имплементация
- Пользовательские утверждения и сессионная персистентность
- Кастомные инструменты для ключевых операций
- Принцип минимальных привилегий в значительной степени
- Комплексная документация и пользовательский опыт

**Достижимо на 60-70%:**
- MCP Dynamic Resolution System (требует значительной разработки, но выполнимо)
- Метапрограммирование агентов (возможно через динамическую генерацию файлов и перезагрузку)
- Полная 12-фазовая система с детальными пре-условиями
- Блокирующие правила с автоматическими пре-проверками
- Детальная система отчетности в требуемом формате

**Требует изменений ядра или имеет фундаментальные ограничения:**
- Полная изоляция состояния на уровне ядра OpenCode
- Автоматическая перезагрузка и горячая замена агентов без перезапуска сессий
- Некоторые аспекты глубокой интеграции с ядром для блокировки определённых действий на низком уровне

### Рекомендации по этапной реализации

**Фаза 1: Базовый каркас (2-3 недели)**
- Реализовать табличный FSM с 4 базовыми состояниями (как в текущем плагине)
- Добавить Pre-Flight проверки через hook `session.created`
- Реализовать базовую систему инструментальных ограничений
- Создать базовую систему промптов и отчетов
- Добавить простую сохранность состояния

**Фаза 2: Расширенный workflow и многоагентность (3-4 недели)**
- Расширить FSM до 8-10 фаз, соответствующих основным этапам Speckit
- Реализовать систему специализированных агентов (планировщик, кодер, рецензент, тестировщик)
- Добавить изолированные сессии рецензирования для проверки артефактов
- Реализовать базовую систему TDD с разделением TEST/CODE
- Добавить систему утверждений пользователя на критических переходах

**Фаза 3: Продвинутые функции и MCP интеграция (4-6 недель)**
- Реализовать MCP Dynamic Resolution System
- Добавить метапрограммирование агентов для динамического создания специализированных агентов
- Внедрить полную 12-фазовую систему workflow с детальными пре-условиями
- Реализовать детальную систему отчетов о делегировании и прогрессе
- Добавить сложные блокирующие правила с автоматическими проверками
- Углубить принцип минимальных привилегий до полной матрицы разрешений

### Заключение

На основе исследования существующих OpenCode плагинов и официальной документации, реализация системы оркестрации, сравнимой по функциональности с Qwen Orchestrator Kit, является **выполнимой** на платформе OpenCode.

Ключевые преимущества подхода:
1. **Плагинная архитектура** позволяет добавлять функциональность без изменений ядра OpenCode
2. **Доказуемые паттерны** существуют в нескольких успешных плагинах (Open Artisan, Micode, Oh My OpenCode)
3. **Гибкость расширения** через кастомные инструменты и хуки событий
4. **Сообщество и экосистема** уже демонстрируют интерес к таким решениям

Главные ограничения:
1. Некоторые глубокие системные интеграции могут требовать изменений ядра
2. Автоматическое метапрограммирование агентов имеет ограничения из-за архитектуры сессий
3. Полная изоляция состояния на уровне процесса сложно достичь без изменений ядра

**Рекомендация**: Начать с реализации базового каркаса (FSM + Pre-Flight + базовые ограничения) и постепенно расширять функциональность в сторону полной совместимости с философией Qwen Orchestrator Kit, используя проверенные паттерны из существующих плагинов как основу.
