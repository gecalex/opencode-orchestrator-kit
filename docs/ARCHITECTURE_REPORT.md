## 1. Обзор проекта

### 1.1 Цель

Создание плагина для OpenCode, реализующего систему оркестрации разработки на основе методологии Speckit/TDD.

### 1.2 Ключевые возможности

| Компонент | Реализация |
|----------|-----------|
| Машина состояний | 10 состояний (0-90) |
| TDD Workflow | TEST/CODE разделение |
| Quality Gates | 6 ворот качества |
| Git Workflow | Автоматические feature-ветки |
| MCP Resolution | 20+ серверов в 12 категориях |
| Блокирующие правила | Автоматические проверки |

## 4. Логика работы

### 4.1 Жизненный цикл задачи

```
1. Получение задачи
         │
         ▼
2. Проверка состояния (state-machine)
         │
         ▼
3. Проверка блокирующих правил
   - state-valid
   - analyst-preconditions
   - coder-tdd
   - git-workflow
         │
    ┌────┴────┐
    │         │
    ▼         ▼
 ОШИБКА   УСПЕХ
    │         │
    ▼         ▼
БЛОК     4. QUALITY GATE 1 (Pre-Execution)
         │
         ▼
    Выполнение задачи
         │
         ▼
    QUALITY GATE 2 (Post-Execution)
         │
         ▼
    Обновление состояния
         │
         ▼
    QUALITY GATE 3 (Pre-Commit) — при git commit
         │
         ▼
    QUALITY GATE 4 (Pre-Merge) — при merge в develop
         │
         ▼
    Завершение
```

### 4.2 TDD Workflow

```
TEST задача ──► Запуск тестов ──► Тесты падают (RED)
                    │
                    ▼
CODE задача ──► Написание кода ──► Тесты проходят (GREEN)
                    │
                    ▼
               Рефакторинг
```

### 4.3 Git Workflow

```
Назначение задачи
        │
        ▼
Создание feature-<task-id> от develop
        │
        ▼
Работа в feature-ветке
        │
        ▼
QUALITY GATE 4 (Pre-Merge)
        │
        ▼
Подтверждение пользователя
        │
        ▼
Merge в develop + Удаление feature-ветки
```

## 5. Философия

### 5.1 Базовые принципы

1. **TDD First!** — тесты перед кодом
2. **Блокирующие правила** — автоматическая проверка
3. **Качество через Gates** — 6 ворот качества
4. **Языковая специализация** — агенты под технологию
5. **Изолированные сессии** — каждая задача в чистой среде

### 5.2 Отличия от Qwen

| Аспект | Qwen | OpenCode |
|--------|------|---------|
| Язык | Shell/Bash | TypeScript |
| Платформа | Qwen CLI | OpenCode |
| MCP | Ручная настройка | Динамическое разрешение |
| Агенты | Bash скрипты | Markdown определения |
| Инструменты | Shell команды | OpenCode tools API |

## 6. Сравнение с Qwen

### 6.1 Что реализовано

#### Qwen (original-qwen-extension)

- ✅ Машина состояний (4 состояния)
- ✅ TDD Workflow
- ✅ Git Workflow (bash скрипты)
- ✅ Quality Gates (5 ворот)
- ✅ MCP Dynamic Resolution (базовый)
- ✅ Блокирующие правила
- ✅ Агенты (shell + markdown)
- ✅ Скиллы (markdown)

#### OpenCode (develop)

- ✅ Машина состояний (10 состояний)
- ✅ TDD Workflow
- ✅ Git Workflow (TypeScript)
- ✅ Quality Gates (6 ворот)
- ✅ MCP Registry (20+ серверов)
- ✅ Блокирующие правила
- ✅ Агенты (markdown)
- ✅ Скиллы (markdown)
- ✅ Матрица разрешений
- ✅ Система отчётности

### 6.2 Как реализовано

| Компонент | Qwen | OpenCode |
|----------|-----|---------|
| Состояния | Bash переменные | TypeScript FSM |
| Переходы | Case/switch | Таблица STATES |
| Агенты | Bash функции | Markdown агенты |
| Инструменты | Shell скрипты | OpenCode API |
| MCP | Bash маппинг | mcp-registry.ts |
| Git | Bash функции | TypeScript модуль |

### 6.3 Почему так реализовано

#### Преимущества TypeScript

1. **Типизация** — ошибки на этапе компиляции
2. **Модульность** — чёткое разделение ответственности
3. **Интеграция** — нативное OpenCode API
4. **Поддержка** — IDE подсказки

#### Недостатки TypeScript

1. **Сложность** — требует компиляцию
2. **Настройка** — больше конфигов

| Метрика | Qwen | OpenCode | Победитель |

| Типизация | ❌ | ✅ | OpenCode |
