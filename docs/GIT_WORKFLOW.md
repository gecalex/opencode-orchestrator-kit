## 🌿 Структура веток

```
main (production)
  ↑
release/vX.X.X
  ↑
develop (development) ← основная ветка для разработки
  ↑
feature/xxx  bugfix/xxx  hotfix/xxx
```

### Ветка `main`
- ✅ Только стабильные релизные версии
- ✅ Мерж **только** из `release/vX.X.X`
- ✅ Теги версий: `v1.0.0`, `v1.1.0`, и т.д.
- ✅ Защищена от прямых коммитов

### Ветка `develop`
- ✅ Основная ветка для разработки
- ✅ Сюда мержатся все feature/bugfix ветки
- ✅ Всегда стабильное состояние для разработки
- ✅ Интеграционное тестирование

### Feature-ветки
- 📍 Создаются от `develop`
- 📍 Именование: `feature/<краткое-описание>`
- 📍 Пример: `feature/user-auth`, `feature/api-integration`
- 📍 После завершения → мерж в `develop` и удаление

### Bugfix-ветки
- 📍 Создаются от `develop`
- 📍 Именование: `bugfix/<краткое-описание>`
- 📍 Пример: `bugfix/login-error`, `bugfix/memory-leak`
- 📍 После исправления → мерж в `develop` и удаление

### Hotfix-ветки
- 📍 Создаются от `main` для критических исправлений
- 📍 Именование: `hotfix/<краткое-описание>`
- 📍 Пример: `hotfix/security-patch`, `hotfix/data-loss`
- 📍 После исправления → мерж в `main` и `develop`

### Release-ветки
- 📍 Создаются от `develop` перед релизом
- 📍 Именование: `release/v<major>.<minor>.<patch>`
- 📍 Пример: `release/v1.0.0`, `release/v1.2.0`
- 📍 После тестирования → мерж в `main` и `develop`
- 📍 Создание тега в `main`

## 📝 Правила коммитов

### Conventional Commits

**Формат сообщения коммита:**
```
<type>: <краткое описание>

[опционально: подробное описание]

[опционально: закрываемые issues]
```

### Типы коммитов

| Тип | Описание | Пример |
|-----|----------|--------|
| `feat` | Новая функция | `feat: добавлена аутентификация пользователя` |
| `fix` | Исправление бага | `fix: исправлена ошибка тайм-аута API` |
| `docs` | Изменения в документации | `docs: обновлена инструкция по установке` |
| `style` | Форматирование, отступы | `style: исправлены отступы в коде` |
| `refactor` | Рефакторинг без изменений функциональности | `refactor: вынесена конфигурация в отдельный модуль` |
| `test` | Добавление/изменение тестов | `test: добавлены тесты для API endpoints` |
| `chore` | Служебные изменения | `chore: обновлены зависимости` |
| `perf` | Улучшение производительности | `perf: оптимизирована загрузка данных` |
| `ci` | Изменения в CI/CD | `ci: добавлен GitHub Actions workflow` |
| `build` | Изменения в системе сборки | `build: обновлена конфигурация webpack` |

### Примеры сообщений коммитов

```bash
feat: добавлена тема Catppuccin Mocha для панели

fix: исправлен таймаут Bluetooth HCI соединения

docs: обновлена документация по хоткеям

refactor: вынесена конфигурация групп в отдельный модуль

chore: обновлены зависимости в requirements.txt

perf: ускорена загрузка изображений на 40%
```

### Правила оформления сообщений

1. **Краткое описание:**
   - Начинается с глагола в повелительном наклонении
   - Первая буква строчная (для consistency с английскими сообщениями)
   - Без точки в конце
   - Максимум 50 символов

2. **Подробное описание:**
   - Отделяется пустой строкой от краткого
   - Объясняет ЧТО и ПОЧЕМУ (не КАК)
   - Строки до 72 символов

3. **Закрываемые issues:**
   - Формат: `Closes #123`, `Fixes #456`
   - Можно указать несколько issues

## 🔄 Процесс разработки

### 1. Создание feature-ветки

```bash
git checkout develop
git pull origin develop
git checkout -b feature/my-new-feature
```

### 2. Внесение изменений

- Делайте изменения в коде
- Перед **каждым коммитом** — ожидание подтверждения
- Коммиты с осмысленными сообщениями
- Частые коммиты (каждые 30-60 минут работы)

### 3. Тестирование в изоляции

**ПРАВИЛО:** Перед внесением изменений в основную конфигурацию

**Методы тестирования:**

#### Синтаксическая проверка
```bash
npx tsc --noEmit
```

#### Проверка импортов
```bash
# Для TypeScript/JS проекты можно проверить через tsc или просто импорт в Node.js
node -e "require('./src/index.ts'); console.log('Import OK')"
```

#### Тестовый запуск
```bash
# Запуск тестов, если они настроены
npm test
# Если тестов нет, можно выполнить базовую проверку линтера
npm run lint 2>/dev/null || echo "Линтер не настроен, пропускаем"
```

#### Резервная копия
```bash
cp src/index.ts src/index.ts.backup
```

### 4. Мерж в develop

```bash
git checkout develop
git merge --no-ff feature/my-new-feature
git branch -d feature/my-new-feature
```

### 5. Подготовка релиза

```bash
git checkout develop
git checkout -b release/v1.0.0
# Финальное тестирование, исправления
git checkout main
git merge --no-ff release/v1.0.0
git tag -a v1.0.0 -m "Release v1.0.0"
git checkout develop
git merge --no-ff release/v1.0.0
git branch -d release/v1.0.0
```

## 🏷️ Теги

### Создание тега после релиза

```bash
git tag -a v1.0.0 -m "Release v1.0.0: начальная стабильная версия"
git push origin v1.0.0
```

### Формат тегов

- `v<major>.<minor>.<patch>` — релизные версии
- `v1.0.0-alpha` — альфа-версии
- `v1.0.0-beta` — бета-версии
- `v1.0.0-rc.1` — релиз-кандидаты

## ⚠️ ПРАВИЛО: Коммит только после тестирования пользователем

**ЗАПРЕЩЕНО:** Создавать коммит без подтверждения пользователя

**АЛГОРИТМ:**

1. Агент вносит изменения
2. Агент тестирует в изоляции (синтаксис, импорты)
3. **Агент показывает изменения пользователю**
4. Пользователь тестирует в рабочей среде
5. **Пользователь подтверждает: "всё работает"**
6. Только тогда агент создаёт коммит

**Пример диалога:**
```
Агент: Изменения внесены. Протестированы в изоляции.
       Готов к коммиту после вашего подтверждения.

Пользователь: Всё работает, создавай коммит.

Агент: [создаёт коммит]
```

**Никогда не создавайте коммит, если:**
- ❌ Пользователь не тестировал
- ❌ Пользователь сказал "подожди"
- ❌ Пользователь сказал "не надо"
- ❌ Вы не получили явного подтверждения

## ✅ ПРАВИЛО: Явное подтверждение коммита

**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО ДЛЯ ВСЕХ АГЕНТОВ**

### Алгоритм коммита

1. Агент вносит изменения
2. Агент показывает изменения пользователю (git diff / preview)
3. **Агент запрашивает явное подтверждение**: "Создать коммит?"
4. Пользователь отвечает: **"да"**, **"коммить"**, **"подтверждаю"**
5. **Только после явного подтверждения** агент создаёт коммит

### Примеры правильного запроса

**✅ ПРАВИЛЬНО:**
```
Агент: Изменения готовы. Создать коммит с сообщением:
       "docs: добавлен отчёт по анализу Qwen Code CLI"

       Файлы:
       - QWEN_CODE_ANALYSIS_REPORT.md (+500 строк)

       Создать коммит?
```

**❌ НЕПРАВИЛЬНО:**
```
Агент: [сразу создаёт коммит без запроса]
```

### Фразы для подтверждения

Пользователь может подтвердить одной из фраз:
- ✅ "да"
- ✅ "коммить"
- ✅ "создавай коммит"
- ✅ "подтверждаю"
- ✅ "всё работает, коммить"

### Запреты

**НИКОГДА не создавайте коммит если:**
- ❌ Пользователь не давал явного подтверждения
- ❌ Пользователь сказал "подожди", "не надо", "сначала покажи"
- ❌ Вы не показали изменения (diff)
- ❌ Вы не запросили подтверждение явно

## ✅ ПРАВИЛО: Подтверждение перед мержем

**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО ДЛЯ ВСЕХ АГЕНТОВ**

### Алгоритм мержа

1. Агент завершает работу в feature-ветке
2. Агент показывает список коммитов для мержа
3. **Агент запрашивает явное подтверждение**: "Сделать мерж в develop?"
4. Пользователь отвечает: **"да"**, **"мержь"**, **"подтверждаю"**
5. **Только после явного подтверждения** агент выполняет мерж

### Примеры правильного запроса

**✅ ПРАВИЛЬНО:**
```
Агент: Feature-ветка завершена. Коммиты для мержа:
       - feat: добавлены 20 Commands
       - feat: добавлены 30 Agent Skills
       - chore: удалён дубликат агента

       Сделать мерж в develop?
```

**❌ НЕПРАВИЛЬНО:**
```
Агент: [сразу делает git merge без запроса]
```

### Фразы для подтверждения

Пользователь может подтвердить одной из фраз:
- ✅ "да"
- ✅ "мержь"
- ✅ "делай мерж"
- ✅ "подтверждаю"
- ✅ "вливай в develop"

### Запреты

**НИКОГДА не делайте мерж если:**
- ❌ Пользователь не давал явного подтверждения
- ❌ Пользователь сказал "подожди", "сначала покажи коммиты"
- ❌ Вы не показали список коммитов
- ❌ Вы не запросили подтверждение явно

## ✅ ПРАВИЛО: Подтверждение перед удалением

**ОБЯЗАТЕЛЬНОЕ ПРАВИЛО ДЛЯ ВСЕХ АГЕНТОВ**

### Что требует подтверждения на удаление

- **Файлы** — любой файл в проекте
- **Директории** — любые папки с содержимым
- **Ветки git** — feature, bugfix, release ветки
- **Агенты/Skills/Commands** — любые компоненты системы
- **Конфигурации** — файлы настроек

### Алгоритм удаления

1. Агент определяет файл/компонент для удаления
2. Агент показывает ЧТО будет удалено и ПОЧЕМУ
3. **Агент запрашивает явное подтверждение**: "Удалить?"
4. Пользователь отвечает: **"да"**, **"удаляй"**, **"подтверждаю"**
5. **Только после явного подтверждения** агент удаляет

### Примеры правильного запроса

**✅ ПРАВИЛЬНО:**
```
Агент: Найдены дубликаты агентов:
       - meta-agent-creator.md (не используется)
       - work_meta_agent_creator.md (используется в документации)

       Удалить meta-agent-creator.md?
```

**✅ ПРАВИЛЬНО:**
```
Агент: Feature-ветка 'feature/add-commands' влита в develop.
       Ветка больше не нужна.

       Удалить feature/add-commands?
```

**❌ НЕПРАВИЛЬНО:**
```
Агент: [сразу удаляет файл без запроса]
```

### Фразы для подтверждения

Пользователь может подтвердить одной из фраз:
- ✅ "да"
- ✅ "удаляй"
- ✅ "удалить"
- ✅ "подтверждаю"
- ✅ "убирай"

### Запреты

**НИКОГДА не удаляйте если:**
- ❌ Пользователь не давал явного подтверждения
- ❌ Пользователь сказал "подожди", "не надо", "сначала покажи"
- ❌ Вы не объяснили ЧТО и ПОЧЕМУ удаляется
- ❌ Вы не запросили подтверждение явно

## 🚫 Запрещённые практики

### Никогда не делайте:

1. **Прямые коммиты в main**
   - ❌ `git checkout main && git commit ...`
   - ✅ Только через release-ветки

2. **Прямые коммиты в develop**
   - ❌ `git checkout develop && git commit ...`
   - ✅ Только через feature-ветки

3. **Force push в общие ветки**
   - ❌ `git push --force origin develop`
   - ✅ Только `git push` или `git push --force-with-lease`

4. **Коммиты без тестирования**
   - ❌ `git commit` без проверки
   - ✅ Тесты → коммит

5. **Огромные коммиты**
   - ❌ Коммит 1000+ строк
   - ✅ Частые небольшие коммиты

6. **Коммиты с чувствительными данными**
   - ❌ API ключи, пароли, токены
   - ✅ Используйте .gitignore и secrets managers

## 🔧 Полезные команды

### Работа с ветками

```bash
# Создать feature-ветку
git checkout develop
git checkout -b feature/my-feature

# Переключиться на ветку
git checkout <branch-name>

# Удалить локальную ветку
git branch -d <branch-name>

# Удалить удалённую ветку
git push origin --delete <branch-name>
```

### Мерж и rebase

```bash
# Мерж ветки
git checkout develop
git merge --no-ff feature/my-feature

# Rebase на develop
git checkout feature/my-feature
git rebase develop

# Отмена rebase
git rebase --abort
```

### Отмена изменений

```bash
# Отменить коммит (сохранить изменения)
git reset --soft HEAD~1

# Отменить коммит (удалить изменения)
git reset --hard HEAD~1

# Отменить изменения в файле
git checkout -- <file>

# Отменить staged изменения
git reset HEAD <file>
```

### Просмотр истории

```bash
# Последние коммиты
git log --oneline -10

# История с графом
git log --oneline --graph --all

# Изменения в коммите
git show <commit-hash>

# Кто изменил строку
git blame <file>
```
