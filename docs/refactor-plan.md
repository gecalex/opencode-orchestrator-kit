# Детальный план рефакторинга Orchestrator Kit для соответствия Qwen Orchestrator Kit

**Цель:** Адаптировать текущую реализацию под философию и workflow оригинального `original-qwen-extension`, устранив выявленные пробелы и сохранив совместимость с архитектурой OpenCode.

## 📜 Основные принципы адаптации

1. **Возврат управления пользователю** – после каждой завершённой фазы оркестратор должен явно вернуть управление и дождаться подтверждения.
2. **Документоориентированность** – `AGENTS.md` является центральным источником истины, аналогично `QWEN.md` в оригинале.
3. **Явное лучше неявного** – все проверки, переходы и действия должны быть прозрачны и требовать подтверждения пользователя там, где это критично.
4. **Минимальные привилегии** – агенты получают доступ только к тем инструментам, которые им действительно нужны.
5. **TDD First!** – тесты пишутся до кода и должны пройти перед переходом к следующей фазе.
6. **Изолированные сессии** – каждая задача выполняется во временной ветке feature/ с чистым состоянием.

## 🏗️ Структура плана

План разбит на **фазы**, соответствующие 12-фазному workflow из оригинала, плюс подготовительные и завершающие шаги.

Каждая фаза включает:
- Цель фазы
- Критерии завершения (конкретный артефакт)
- Требуемые подтверждения
- Связанные задачи

## 🗂️ Фазы работ

### Фаза 0: Подготовка и базовая инфраструктура
**Цель:** Подготовить репозиторий к адаптации: добавить недостающие документы, улучшить логирование, подготовить шаблоны.
**Критерии завершения:**
- Существует шаблон `AGENTS.md` с разделом "Философия и принципы"
- Есть функция вывода текущего состояния и следующих шагов
- Есть шаблоны диалогов подтверждения для всех фаз
**Задачи:** (см. `tasks.md` блок 0)

### Фаза 1: Инициализация проекта (соответствие Правилу 1.1)
**Цель:** Обеспечить, чтобы инициализация Git и базовых файлов требовала явного подтверждения пользователя и возвращала управление после каждого шага.
**Критерии завершения:**
- Пользователь явно подтверждает создание .git и ветки develop
- Пользователь явно подтверждает создание .gitignore и README.md
- После инициализации оркестратор возвращает управление и предлагает создать конституцию
**Задачи:** (см. `tasks.md` блок 1)

### Фаза 2: Создание конституции проекта
**Цель:** Делегировать создание `PROJECT.md` специализированному агенту `speckit-constitution-agent` с обязательным диалогом подтверждения.
**Критерии завершения:**
- Создан файл `PROJECT.md` с метаданными (версия, дата, статус)
- Пользователь подтвердил содержание конституции
- Оркестратор возвратил управление
**Задачи:** (см. `tasks.md` блок 2)

### Фаза 3: Создание спецификаций модулей
**Цель:** Делегировать создание спецификаций в директорию `SPEC/` агенту `speckit-specify-agent`.
**Критерии завершения:**
- В директории `SPEC/` созданы файлы спецификаций для каждого модуля
- Каждая спецификация содержит входы, выходы, интерфейсы, зависимости
- Пользователь подтвердил набор спецификаций
**Задачи:** (см. `tasks.md` блок 3)

### Фаза 4: Анализ противоречий в спецификациях
**Цель:** Выявить и разрешить конфликты между спецификациями.
**Критерии завершения:**
- Создан файл `CONFLICTS_RESOLUTION.md`
- Все выявленные противоречия документированы и предложены пути решения
- Пользователь подтвердил анализ
**Задачи:** (см. `tasks.md` блок 4)

### Фаза 5: Планирование реализации (общий план)
**Цель:** Создать общий план реализации с учётом зависимостей, оценкой усилий и рисками.
**Критерии завершения:**
- Создан файл `IMPLEMENTATION_PLAN.md`
- План включает порядок реализации модулей, оценку в чел-часах/story points, технические риски и стратегии их mitigation
- Пользователь подтвердил план
**Задачи:** (см. `tasks.md` блок 5)

### Фаза 6: Разбивка на задачи с зависимостями
**Цель:** Разбить модули на атомарные задачи (max 8 часов) с чёткими критериями принятия и зависимостями.
**Критерии завершения:**
- Создан файл `TASKS.md` с таблицей задач (ID, описание, критерии принятия, зависимости, тип агента)
- Все задачи атомарны и независимы (где возможно)
- Пользователь подтвердил разбивку
**Задачи:** (см. `tasks.md` блок 6)

### Фаза 7: Анализ пробелов в требованиях
**Цель:** Сравнить текущие спецификации с оригинальными TZ.md и выявить неочевидные сценарии использования.
**Критерии завершения:**
- Создан файл `REQUIREMENTS_GAP_ANALYSIS.md`
- Выявлены неочевидные сценарии, пробелы в покрытии нефункциональных требований (производительность, безопасность)
- Сформулированы уточняющие вопросы для стейкхолдеров
- Пользователь подтвердил анализ
**Задачи:** (см. `tasks.md` блок 7)

### Фаза 8: Назначение агентов на задачи
**Цель:** Назначить ответственных агентов на каждую задачу с учётом их специализации и загрузки.
**Критерии завершения:**
- Создан файл `ASSIGNMENTS.md` с соответствием задачи → агент
- Учтена текущая загрузка агентов и доступность необходимых навыков
- Для каждого кодерского задачи назначен ответственный рецензент
- Пользователь подтвердил назначения
**Задачи:** (см. `tasks.md` блок 8)

### Фаза 9: Создание TEST/CODE разделения (TDD First!)
**Цель:** Для каждой кодерской задачи создать парную тестовую задачу, которая выполняется первой.
**Критерии завершения:**
- В `TASKS.md` отмечены тестовые задачи с префиксом `TEST:`
- Тестовые задачи имеют чёткое техническое задание и выбран тестовый фреймворк
- Тестовые задачи получают высший приоритет в планировании
- Пользователь подтвердил разделение
**Задачи:** (см. `tasks.md` блок 9)

### Фаза 10: TDD Координация (языковая специализация)
**Цель:** Назначить тестовых агентов, специализированных на языке проекта, и настроить изолированные среды для выполнения тестов.
**Критерии завершения:**
- Создан файл `TEST_ENVIRONMENT_SETUP.md`
- Настроены контейнеры/виртуальные окружения для запуска тестов
- Установлены правила написания тестовых данных и моков
- Тестовые агенты имеют доступ к необходимым библиотекам (pytest, Jest и т.д.)
- Пользователь подтвердил настройку
**Задачи:** (см. `tasks.md` блок 10)

### Фаза 11: Разработка кода под тесты
**Цель:** Написание минимума кода для прохождения конкретного теста с последующим рефакторингом без регрессии.
**Критерии завершения:**
- Код пишется только для прохождения failing теста (RED → GREEN)
- После зелёного выполняется рефакторинг с сохранением прохождения всех тестов
- Следуется принципам чистого кода и соглашениям проекта
- Ежечасный коммит с описательными сообщениями
- Пользователь подтверждает каждый цикл RED-GREEN-REFACTOR
**Задачи:** (см. `tasks.md` блок 11)

### Фаза 12: Выполнение задач и интеграция
**Цель:** Выполнить все задачи, запустить полный набор тестов, подготовить артефакты релиза.
**Критерии завершения:**
- Запущен полный тест набор (юнит, интеграционный, системный)
- Проверено соответствие нефункциональным требованиям
- Подготовлены артефакты релиза (сборка, документация, заметки о релизе)
- Выполнено финальное рецензирование интегрированного решения
- Пользователь подтвердил готовность к релизу
**Задачи:** (см. `tasks.md` блок 12)

### Фаза 13: Интеграция систем проверок и аналитики
**Цель:** Объединить pre-flight проверки из оригинала с системой quality gates, добавить аналитику и обратную связь.
**Критерии завершения:**
- Единая точка входа для проверки готовности к любой операции (`check-readiness`)
- Сохранена возможность автоматического прохождения проверок для CI
- Руководящий принцип – ручной режим как дефолтный для максимальной прозрачности
- Введена система аналитики вызовов агентов (время, успех/провал)
- Генерируются отчёты в стиле `analytics/reports/agent-call-analysis-*.md`
**Задачи:** (см. `tasks.md` блок 13)
