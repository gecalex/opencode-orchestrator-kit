## ⚙️ Логика инициализации (гибридная)

**Основная проблема решена:** Баг OpenCode #5894 - `tool.execute.before` не работает для subagent  

**Реализовано:**
1. **Авто-инициализация при загрузке плагина** (через plugin config hook)
   - Выполняется один раз при старте плагина
   - Создаёт `.git`, `.gitignore`, `README.md`, `AGENTS.md`
   - Логирует в `.opencode/orchestrator.log`

2. **Fallback механизм** (в `tool.execute.before`)
   - Если авто-инициализация не сработала - пробует при первом tool call
   - Предотвращает полную неинициализацию проекта

3. **Кастомный tool** `orchestrator-init`
   - Позволяет ручную инициализацию с параметрами
   - Может выполнять принудительную переинициализацию

## 🔄 Паттерн выполнения задач

Каждая задача следует циклу:
1. **Получение назначения** - agent получает задачу с критериями принятия
2. **Проверка пре-условий** - автоматическая проверка готовности
3. **Выполнение в изолированной сессии** - временная ветка feature/
4. **Самотестирование** - запуск релевантных тестов
5. **Подготовка отчёта** - структурированный markdown отчёт
6. **Запрос подтверждения** - показ изменений и запрос на мерж
7. **Интеграция** - автоматический мерж в develop после подтверждения

## 🛡️ Блокирующие правила (обязательные для всех агентов)

### Аварийный тормоз
Перед запуском любого агента проверяется:
- Соответствие текущего состояния workflow разрешенному для типа агента
- Наличие явного подтверждения пользователя на критических переходах
- Выполнение всех pre-condition проверок для текущей фазы
- Доступность необходимых MCP серверов

### Правило пре-условий
- **Аналитики:** доступность исходных данных и спецификаций
- **Кодеры:** наличие написанных и прошедших тестов (TDD First!)
- **Рецензенты:** наличие артефакта для рецензирования и понимания требований
- **Тестировщики:** наличие тестовых сценариев и среды выполнения

### Правило подтверждения пользователя
Требуется на переходах:
- Анализ → Планирование (3→4)
- Планирование → Выполнение (6→7)
- Написание тестов → Написание кода (9→10)
- Любое изменение критической архитектуры или БД
- Удаление файлов или компонентов системы

## 🔌 MCP Dynamic Resolution System

**Принцип работы:**
1. При начале фазы реализации система анализирует:
   - Требуемые технологии из спецификаций модулей
   - Текущий стек проекта из конституции
   - Доступность локальных инструментов и SDK
2. Для недостающих компонентов выполняется поиск MCP серверов
3. Выбранные MCP серверы автоматически скачиваются, конфигурируются и регистрируются
4. Кеширование в `.opencode/mcp-cache/` для повторного использования  

**Текущие MCP серверы в opencode.json:**
- **filesystem** - локальный доступ к файлам
- **git** - git операции
- **memory** - хранение контекста
- **github** - GitHub API
- **context7** - удалённая документация
- **searxng** - поиск (временно отключён из-за проблем)
- **sentry** - мониторинг ошибок (отключён)
