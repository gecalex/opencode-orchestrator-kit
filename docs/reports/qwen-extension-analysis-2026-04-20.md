# Анализ ветки original-qwen-extension и сравнение с текущей реализацией
**Дата:** 2026-04-20  
**Цель:** Выявление пробелов, минусов, просчетов и искажений философии при адаптации Qwen Orchestrator Kit для OpenCode  

---

## 🔍 Вводная информация

**Исходный проект:** `original-qwen-extension` - реализация оркестратора для Qwen Code CLI  
**Целевой проект:** `opencode_orc_kit` - адаптация для OpenCode  
**Основная проблема:** Различия в архитектуре между Qwen Code CLI и OpenCode приводят к серьёзным трудностям в адаптации  

---

## 📊 Сравнительная таблица ключевых различий

| Аспект | Qwen Code CLI (original) | OpenCode (наш проект) | Последствия для адаптации |
|--------|--------------------------|----------------------|---------------------------|
| **Архитектура агентов** | Глубокая иерархия специализированных агентов (см. agents/*) | Плоская структура агентов в opencode.json | Непрямое соответствие агентов |
| **Механизм инициализации** | `work_dev_project_initializer` - отдельный агент, возвращает управление | Авто-инициализация при загрузке плагина + fallback | Разная точка входа в workflow |
| **Состояния проекта** | 10 состояний (пустой → релиз-готов) | 10 состояний (аналогично) | Совместимы, но разная нумерация |
| **Механизм подтверждения** | Явные запросы пользователя после каждой фазы | Предупреждения в логах + качественные гейты | Нужен гибридный подход |
| **Хранение контекста** | Встроенные скрипты и переменные окружения | Система сохранения контекста (.opencode/context/) | Разные подходы к сохранению состояния |
| **Методология** | Чистый Speckit + TDD | Speckit/TDD с адаптациями под OpenCode | Необходима гармонизация |
| **Git workflow** | Специализированные скрипты в `$EXTENSION_DIR/scripts/git/` | Интеграция через git-workflow.ts и стандартные Git команды | Требуется абстракция |

## 🧩 Детальный анализ компонентов original-qwen-extension

### 1. **QWEN.md - главный файл инструкций (1416 строк)**

#### Ключевые особенности:
- **Жёсткая иерархия правил**: Запрещено запускать любых агентов до инициализации через `work_dev_project_initializer`
- **Явные чек-листы пре-условий**: Обязательная проверка 6+ пунктов перед каждым агентом
- **Автоматический аварийный тормоз**: Блокирующий скрипт `pre-flight-check.sh` проверяет текущую фазу
- **Структурированный быстрый старт**: После каждой фазы агент обязан показать результат и запросить подтверждение
- **Детальный workflow с фазами**: 13 фаз от анализа до сохранения контекста

#### Проблемы при адаптации к OpenCode:
1. **Конфликт с архитектурой OpenCode**: Qwen требует, чтобы агенты не запускались до инициализации, но в OpenCode плагин загружается автоматически
2. **Избыточность подтверждений**: Qwen требует подтверждения пользователя после КАЖДОЙ фазы, что замедляет работу
3. **Жёсткая привязка к filesystem**: Скрипты в `$HOME/.qwen/extensions/` не переносимы в OpenCode
4. **Отсутствие интеграции с качественными гейтами**: В Qwen своя система проверок, в OpenCode - 5 gate системы

### 2. **Агенты и их назначения**

#### Ключевые агенты в оригинале:
- **work_dev_project_initializer** - инициализация Git, .gitignore, базовые навыки
- **speckit-constitution-agent** - создание конституции проекта
- **speckit-specify-agent** - создание спецификаций модулей
- **speckit-plan-agent** - создание плана реализации
- **speckit-tasks-agent** - разбивка на задачи
- **work_* агенты** - исполнители (разработчики, тестировщики и т.д.)
- **orc_* агенты** - координаторы и аналитики

#### Особенности реализации:
- Каждый агент имеет чётко указанные инструменты (tools:)
- Агенты имеют цветовое кодирование для визуальной идентификации
- Явно указаны зависимости и порядок выполнения
- Используют навыки (skills:) для повторяемой логики

#### Проблемы при адаптации:
1. **Несоответствие навыков**: Навыки в Qwen (speckit-constitution, init-project и т.д.) не совпадают с навыками в нашем проекте
2. **Инструменты разного уровня доступа**: Некоторые инструменты в оригинале могут быть недоступны в OpenCode из-за политики безопасности
3. **Отсутствие механизма делегирования**: В оригинале агенты явно вызывают друг друга через task, в нашей реализации есть auto-делегирование

### 3. **Структура проекта и артефакты**

**Оригинальная структура:**
```
project/
├── QWEN.md              # Главный файл инструкций
├── PROJECT.md           # Конституция (создаётся speckit-constitution-agent)
├── SPEC/                # Директория спецификаций
├── TASKS.md             # Задачи
├── REPORTS/             # Отчёты после операций
├── analytics/           # Аналитика вызовов агентов
├── workouts/            # TODO: не понятно
└── agents/              # Все агенты в виде .md файлов
```

**Наша структура:**
```
project/
├── AGENTS.md            # Инструкции для оркестратора
├── PROJECT.md           # Конституция
├── SPEC/                # Директория спецификаций
├── TASKS.md             # Задачи
├── REPORTS/             # Отчёты
├── src/                 # Исходный код плагина (TypeScript)
└── .opencode/           # Ресурсы плагина
```

#### Критические различия:
1. **Расположение инструкций**: В оригинале - отдельный QWEN.md, в нашей реализации - часть AGENTS.md
2. **Хранение логики**: В оригинале - agents/ как набор инструкций, в нашей реализации - agents/ как исполняемый код
3. **Отсутствие аналитики**: В оригинале есть детальная система отслеживания вызовов агентов, в нашей реализации - базовые отчёты
4. **Разная парадигма расширяемости**: В оригинале - добавление новых агентов через .md файлы, в нашей реализации - через TypeScript классы

### 4. **Механизм инициализации и workflow**

**Оригинальный процесс инициализации:**
1. Пользователь вручную запускает: `task '{"subagent_type": "work_dev_project_initializer", "prompt": "Инициализируй проект"}'`
2. Агент выполняет:
   - Проверка расширения
   - Инициализация Git
   - Создание .gitignore
   - Запуск skill: init-project
   - Запуск skill: analyze-state
   - Запуск skill: pre-flight
   - **ВОЗВРАЩАЕТ управление оркестру** (не продолжает автоматически!)
3. Пользователь подтверждает и переходит к следующему шагу

**Наш процесс инициализации:**
1. Плагин загружается автоматически при старте OpenCode
2. Выполняется `autoInitOnPluginLoad()`:
   - Создаёт .git, .gitignore, README.md, AGENTS.md
   - Обновляет state machine до состояния 2
3. Возможен fallback через `tool.execute.before`

#### Проблемы в текущей реализации:
1. **Нарушение принципа "ВЕРНЁТ управление"**: Наша авто-инициализация продолжает работу автоматически, не возвращая управление пользователю
2. **Отсутствие обязательного подтверждения**: В оригинале после инициализации агент ОБЯЗАН спросить подтверждение, у нас этого нет
3. **Неполная имитация оригинального workflow**: Мы пропускаем этап явного возврата управления и диалога с пользователем
4. **Конфликт с блокирующими правилами**: Наша авто-инициализация может нарушать пре-условия, так как выполняется без проверки готовности

### 5. **Качественные гейты и система проверок**

**В original-qwen-extension:**
- Пре-условия проверяются через чек-листы в markdown
- Есть специальный скрипт `pre-flight-check.sh` для аварийного тормоза
- После каждой фазы есть встроенная валидация результатов

**В нашей реализации:**
- 5-уровневая система quality gates
- Автоматические проверки перед и после выполнения инструментов
- Блокирующие правила как отдельный слой защиты

#### Проблемы согласования:
1. **Дублирование проверок**: Наша качественная gate система может конфликтовать с pre-flight проверками из оригила
2. **Разная философия проверок**: В оригинале проверки - это диалог с пользователем, у нас - автоматические проверки
3. **Отсутствие явных артефактов проверок**: В оригинале после каждой фазы остаются отчётные файлы, у нас проверки проходят "незаметно"

## ⚠️ Выявленные пробелы и минусы в текущей реализации

### Критические пробелы:
1. **Нарушение принципа возврата управления** (Правило 1.1 из QWEN.md)
   - **Текущая реализация**: Авто-инициализация продолжает работу без возврата управления
   - **Оригинал**: После инициализации агент ОБЯЗАТЕЛЬНО возвращает управление
   - **Последствия**: Пользователь лишён возможности осознанно подтвердить каждый этап

2. **Отсутствие обязательного диалога подтверждения** (Раздел 2.2 из QWEN.md)
   - **Текущая реализация**: Предупреждения идут в логи, нет блока диалога
   - **Оригинал**: После каждой фазы агент ДОЛЖЕН показать результат и спросить "Продолжить? (да/нет)"
   - **Последствия**: Снижение прозрачности и контроля пользователя над процессом

3. **Неполная имитация workflow фаз** (Раздел 6 из QWEN.md)
   - **Текущая реализация**: Мы реализовали паттерн выполнения задач, но не 12-фазный workflow
   - **Оригинал**: Чётко разделенные фазы с конкретными артефактами и действиями
   - **Последствия**: Потеря структурированности и предсказуемости процесса

4. **Конфликт систем проверок**
   - **Текущая реализация**: 5 gate качества + блокирующие правила
   - **Оригинал**: Pre-flight проверки + чек-листы пре-условий
   - **Последствия**: Возможны ложные срабатывания или пропуски проверок

### Просчеты в логике:
1. **Нарушение атомарности фаз**: В оригинале каждая фаза создаёт конкретный артефакт (PROJECT.md, SPEC/*.md и т.д.) и явно завершается
2. **Отсутствие версии и метаданных**: В оригинале каждый документ имеет версию, дату, статус - в нашей реализации этого нет
3. **Несоответствие инструментов агентов**: Агенты в оригинале имеют чётко указанный набор инструментов, у нас этот механизм слабо реализован
4. **Отсутствие аналитики и обратной связи**: В оригинале есть система отслеживания вызовов агентов, у нас базовые отчёты

### Искажения философии:
1. **Потеря принципа "Явное лучше неявного"**: В оригинале всё явно прописано и требует действий пользователя, у нас много автоматических процессов
2. **Смещение акцента с диалога на автоматизация**: Оригинал делает акцент на осознанном прогрессе через диалог, мы - на скорости через автоматизацию
3. **Разделение ответственности**: В оригинале чётко разведены роли инициализаторов, конституторов, спецификаторов - у нас это менее выражено

## 💡 Рекомендации по элегантному решению конфликта архитектур

### Принцип адаптации:
**Не копировать, а транслировать принципы** - сохранить философию и workflow оригинала, реализовав их через механизмы OpenCode

### Конкретные рекомендации:

#### 1. **Восстановить принцип возврата управления**
- Изменить `autoInitOnPluginLoad()` так, чтобы она только проверяла и подготавливала окружение, но **НЕ** переходила к следующим шагам автоматически
- Добавить явный этап подтверждения после инициализации через диалог с пользователем
- После подтверждения - перейти к созданию конституции через отдельный вызов агента

#### 2. **Реализовать обязательные диалоги подтверждения**
- После каждой завершённой фазы (анализ, конституция, спецификации и т.д.) показывать структурированный отчёт
- Блокировать дальнейшее выполнение до получения явного подтверждения пользователя ("да"/"нет")
- Использовать механизм `client.session.prompt()` для создания блокирующих диалогов

#### 3. **Восстановить 12-фазный workflow как основу**
- Чётко разделить работу на 12 фаз согласно оригиналу:
  1. Анализ состояния проекта
  2. Создание конституции проекта
  3. Создание спецификаций модулей
  4. Анализ противоречий в спецификациях
  5. Планирование реализации
  6. Разбивка на задачи с зависимостями
  7. Анализ пробелов в требованиях
  8. Назначение агентов на задачи
  9. Создание TEST/CODE разделения (TDD First!)
  10. TDD Координация
  11. Разработка кода под тесты
  12. Выполнение задач и интеграция
- Каждая фаза должна создавать конкретный артефакт и требовать подтверждения

#### 4. **Создать абстракцию над проверками пре-условий**
- Вместо прямого копирования скриптов из оригинала, создать адаптер:
  - `pre-flight-check.sh` → TypeScript функция, проверяющая текущую фазу через state machine
  - Чек-листы пре-условий → Автоматические проверки + опциональный диалог подтверждения
- Интегрировать с существующей системой quality gates, а не дублировать её

#### 5. **Сохранить дух документоориентированности**
- Сделать AGENTS.md настоящим центральным документом, как QWEN.md в оригинале
- Позволить агентам читать и интерпретировать этот документ для принятия решений
- Использовать его как источник истины о текущем состоянии и следующих шагах

#### 6. **Реализовать механизм явного делегирования**
- Вместо автоматического делегирования, требовать явного подтверждения пользователя на переходе между типами агентов
- Сохранить возможность автоматического выполнения для опытных пользователей через настройку

#### 7. **Добавить систему аналитики и обратной связи**
- Следить за вызовами агентов, временем выполнения, успехами/провалами
- Генерировать отчёты в стиле оригинала (analyics/reports/agent-call-analysis-*.md)
- Использовать эту аналитику для улучшения workflow
