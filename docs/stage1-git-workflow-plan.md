# План этапа 1: Интеграция Git workflow

## Цель
Интегрировать полноценный Git workflow в плагин OpenCode Orchestrator Kit, включая:
- Автоматическое создание feature-веток при назначении задач
- Блокировку коммита при отсутствии feature-ветки
- Обязательное подтверждение пользователя перед мержем
- Интеграцию Quality Gate 4 (Pre-Merge) в процесс мержем
- Автоматическое удаление feature-ветки после успешного мержем

## Задачи

### 1. Блокирующее правило для Git workflow проверки
- Добавить проверку в `blocking-rules.ts`: перед выполнением любой работы агента проверять, что мы находимся в feature/bugfix/hotfix ветке (кроме начальной инициализации)
- Интегрировать эту проверку в `index.ts` в `tool.execute.before` хуке для соответствующих инструментов
- Добавить соответствующее сообщение об ошибке и рекомендацию

### 2. Автоматическое создание feature-веток при назначении задач
- Изменить `orchestrator.ts`: при назначении задачи на агента автоматически создавать feature-ветку от develop
- Использовать существующий метод `createFeatureBranch` из `gitWorkflow`
- Ветка должна называться `feature/<task-id>-<краткое-описание>` или аналогично
- Сохранять информацию о текущей ветке в контексте сессии

### 3. Блокировка коммита при отсутствии feature-ветки
- Улучшить существующую проверку в `index.ts` (строка 76-82): добавить проверку, что мы находимся в feature/bugfix/hotfix ветке перед выполнением `git commit`
- Если проверка не пройдена, блокировать коммит и показывать инструкцию по созданию ветки
- Интегрировать с блокирующими правилами для единой точки проверки

### 4. Интеграция Quality Gate 4 (Pre-Merge) в процесс мержем
- Изменить функцию `mergeToDevelop` в `git-workflow.ts`: перед мержем запускать Gate 4 (Pre-Merge)
- Если Gate 4 не пройден, блокировать мерж и показывать детальный отчёт
- Добавить возможность принудительного мержем только с явным подтверждением пользователя (для исключительных случаев)

### 5. Реализация обязательного подтверждения пользователя перед мержем
- Изменить функцию `mergeToDevelop` в `git-workflow.ts`: перед мержем показывать список коммитов и запрашивать подтверждение
- Использовать существующую систему подтверждения пользователя из `user-approval.ts` или создать аналогичную
- Только после явного подтверждения выполнять мерж

### 6. Автоматическое удаление feature-ветки после успешного мержем
- Улучшить существующую функцию `mergeToDevelop` в `git-workflow.ts`: после успешного мержем и push автоматически удалять локальную feature-ветку
- Добавить опцию сохранения ветки для отладки (по умолчанию отключена)
- Логировать удаление ветки в журнал сессий

## Связанные файлы
- `src/index.ts` - основной хук плагина
- `src/orchestrator.ts` - основной класс оркестратора
- `src/git-workflow.ts` - Git операции
- `src/blocking-rules.ts` - блокирующие правила
- `src/quality-gates.ts` - контрольные точки качества
- `src/user-approval.ts` - система подтверждения пользователя
- `src/types.ts` - типы данных

## Критерии готовности
- [ ] Все Git операции выполняются в изолированных feature-ветках
- [ ] Коммит невозможен вне feature/bugfix/hotfix ветки
- [ ] Мерж в develop возможен только после прохождения Gate 4 и подтверждения пользователя
- [ ] Feature-ветка автоматически удаляется после успешного мержем
- [ ] Все изменения покрыты тестами (если применимо)
- [ ] Документация обновлена с инструкциями по использованию
