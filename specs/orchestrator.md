# Спецификация модуля orchestrator.ts

## 1. Обзор

Модуль orchestrator.ts является главным входом в плагин Orchestrator Kit. Он координирует работу всех компонентов системы, управляет workflow выполнения задач и обеспечивает соблюдение методологии Speckit.

## 2. Функциональность

- Инициализация плагина и загрузка всех необходимых модулей
- Координация работы агентов и скиллов согласно текущему состоянию проекта
- Управление workflow выполнения задач (анализ, планирование, выполнение, ревью)
- Обеспечение соблюдения правил подтверждения пользователя на критических переходах
- Интеграция с системой Quality Gates для проверки качества на каждом этапе
- Управление изолированными сессиями выполнения задач в feature-ветках
- Координация TDD workflow (TEST/CODE разделение)
- Обработка событий сессии и сохранение контекста
- Предоставление кастомных инструментов (`speckit-*`) для использования агентами

## 3. Интерфейсы

### Входы:
- Комманды от пользователя через opencode CLI
- События сессии (создание, простой, компактация, ошибка)
- Запросы на выполнение конкретных операций (анализ состояния, инициализация и т.д.)
- Подтверждения пользователя на критических переходах
- Результаты выполнения агентов и скиллов

### Выходы:
- Ответы пользователю с информацией о выполненных операциях
- Статусы выполнения workflow и переходов между состояниями
- Отчеты о прогрессе проекта и выполненных задачах
- Уведомления о необходимости действий (подтверждение, исправление ошибок и т.д.)
- Обновленное состояние проекта в хранилище

### Внутренние методы:
- `initializePlugin(): Promise<void>` - инициализация всех модулей плагина
- `handleCommand(command: string, args: any[]): Promise<CommandResult>` - обработка команд пользователя
- `transitionState(targetState: StateCode, userConfirmed: boolean): Promise<StateTransitionResult>` - управление переходами состояний
- `executeWorkflowStep(step: WorkflowStep, taskId: string): Promise<WorkflowStepResult>` - выполнение шага workflow
- `validatePreConditions(agentType: AgentType): Promise<PreConditionResult>` - проверка пре-условий перед запуском агента
- `notifyUser(message: string, options?: NotificationOptions): void` - уведомление пользователя
- `saveSessionContext(): Promise<void>` - сохранение контекста сессии
- `loadSessionContext(): Promise<SessionContext>` - загрузка контекста сессии

## 4. Ограничения

- Все критические переходы требуют явного подтверждения пользователя
- Плагин должен обеспечивать строгое следование методологии TDD First!
- Выполнение задач должно происходить в изолированных feature-ветках
- Каждая операция должна проходить соответствующие Quality Gates
- Плагин должен блокировать выполнение операций, если пре-условия не выполнены
- Должен обеспечивать обратную совместимость с предыдущими версиями
- Все ответы пользователю должны быть на русском языке

## 5. Внешние интеграции

- Интегрируется со всеми 16 агентами через систему делегирования задач
- Взаимодействует с 9 скиллами для специализированных операций
- Подключается к системе MCP Dynamic Resolution для доступа к внешним серверам
- Использует систему разрешений (permissions.ts) для контроля доступа
- Взаимодействует с блокирующими правилами (blocking-rules.ts) для проверки переходов
- Интегрируется с модулем состояния (state-machine.ts) для определения текущего состояния
- Взаимодействует с workflow-модулями (pre-flight, tdd-workflow, git-workflow, quality-gates, reporting)
- Подключается к системе управления сессиями (session-hooks, review-sessions)
- Использует систему генерации отчетов (reporting.ts)

## 6. Требования

1. Должен обеспечивать точку входа для всех операций плагина через opencode CLI
2. Должен управлять состоянием проекта через интеграцию с state-machine.ts
3. Должен координировать работу агентов согласно текущему состоянию и назначениям
4. Должен обеспечивать соблюдение всех блокирующих правил и пре-условий
5. Должен управлять workflow выполнения задач (анализ → планирование → выполнение → ревью → интеграция)
6. Должен обеспечивать TDD First! подход (тесты перед кодом)
7. Должен управлять изолированными сессиями в feature-ветках
8. Должен интегрироваться с системой Quality Gates для проверки качества
9. Должен предоставлять кастомные инструменты speckit-* для использования в агентах и скиллах
10. Должен сохранять и восстанавливать контекст сессии между операциями
11. Должен обеспечивать явное подтверждение пользователя на всех критических переходах
12. Должен генерировать структурированные отчеты о выполненных операциях
