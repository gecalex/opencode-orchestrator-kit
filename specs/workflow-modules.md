# Спецификация модуля workflow-modules

## 1. Обзор

Модуль workflow-modules объединяет пять ключевых workflow-компонентов Orchestrator Kit: pre-flight проверки, TDD workflow, Git workflow, Quality Gates и систему отчетов. Эти компоненты обеспечивают соблюдение методологии Speckit на всех этапах разработки.

## 2. Компоненты

### 2.1 Pre-Flight проверки (pre-flight.ts)
- Проверка готовности проекта перед фазой разработки
- Проверяет Git, ветку develop, .gitignore, конституцию, агентов и скиллы
- Блокирует запуск агентов, если пре-условия не выполнены

### 2.2 TDD Workflow (tdd-workflow.ts)
- Обеспечивает обязательное разделение TEST/CODE фаз
- Тестовые задачи создаются и выполняются ПЕРЕД кодовыми задачами
- Блокирует переход к кодовой задаче до прохождения тестов
- Управляет связью между TEST и CODE задачами

### 2.3 Git Workflow (git-workflow.ts)
- Управление Git операциями: создание feature-ветки, pre-commit review, merge, conventional commits
- Обеспечивает работу в изолированных feature-ветках
- Автоматический возврат к develop после завершения задачи
- Требует подтверждения пользователя перед мержем

### 2.4 Quality Gates (quality-gates.ts)
- Система из 6 ворот качества:
  1. Pre-Execution (синтаксическая валидность)
  2. Post-Execution (тестовое покрытие ≥ 80%)
  3. Pre-Commit (безопасность линтинг)
  4. Pre-Merge (интеграционные проверки)
  5. Pre-Implementation (соответствие спецификациям)
  6. MCP Check (доступность MCP серверов)
- Блокирует переход к следующей фазе, если ворота не пройдены

### 2.5 Отчеты (reporting.ts)
- Генерация структурированных отчетов после каждой операции
- Сохранение отчетов в директорию REPORTS/ с маркировкой даты и типа операции
- Формат отчета: Summary, Changes Made, Results, Issues & Risks, Attachments
- Предоставляет данные для отслеживания прогресса проекта

## 3. Интерфейсы

### Входы (общие для всех компонентов):
- Текущее состояние проекта (от state-machine.ts)
- Команды от оркестратора
- Данные о выполненных операциях
- Подтверждение пользователя (где требуется)
- Результаты предыдущих проверок

### Выходы (общие для всех компонентов):
- Статус выполнения операции (успех/неудача)
- Обновленное состояние проекта (если применимо)
- Отчеты о выполненной работе
- Уведомления о необходимости действий
- Данные для следующих этапов workflow

### Взаимодействие между компонентами:
- Pre-Flight проверяет готовность перед запуском любого агента
- TDD Workflow обеспечивает, что тестовые задачи выполняются перед кодовыми
- Git Workflow управляет изолированными сессиями и мержем в develop
- Quality Gates проверяются перед каждым критическим переходом
- Reporting собирает данные от всех компонентов и генерирует отчеты

## 4. Ограничения

- Pre-Flight проверки должны пройти перед запуском любого агента
- TDD First! принцип обязателен: тестовые задачи перед кодовыми
- Все операции должны происходить в изолированных feature-ветках
- Каждая операция должна пройти соответствующие Quality Gates
- Мерж в develop возможен только после прохождения всех ворот и подтверждения пользователя
- Отчеты должны генерироваться после каждой завершенной операции
- Все компоненты должны взаимодействовать через четко определенные интерфейсы

## 5. Внешние интеграции

- Взаимодействуют с оркестратором (orchestrator.ts) для получения команд и отправки результатов
- Интегрируются с state-machine.ts для определения разрешенных операций в текущем состоянии
- Используют blocking-rules.ts для проверки пре-условий
- Взаимодействуют с permissions.ts для проверки разрешений на операции
- Интегрируются с agent-system для назначения и выполнения задач конкретными агентами
- Взаимодействуют с mcp-integration для проверки доступности MCP серверов (Quality Gate 6)
- Интегрируются с session-management для обработки событий сессии
- Используют файловые операции для чтения/записи конфигурации и состояния

## 6. Требования

1. Pre-Flight компонент должен проверять все критические предусловия перед запуском агентов
2. TDD Workflow компонент должен обеспечивать обязательное создание и выполнение тестовых задач перед кодовыми
3. Git Workflow компонент должен управлять созданием feature-веток, коммитами, мерджем и соблюдением conventional commits
4. Quality Gates компонент должен реализовать все 6 ворот качества с четкими критериями прохождения
5. Reporting компонент должен генерировать структурированные отчеты после каждой операции в требуемом формате
6. Все компоненты должны обеспечивать явное подтверждение пользователя на критических переходах
7. Все компоненты должны работать в изолированных сессиях (feature-ветки)
8. Все компоненты должны интегрироваться с оркестратором и другими модулями системы
9. Все компоненты должны следовать принципу минимальных привилегий
10. Все компоненты должны быть независимы от конкретной реализации и легко тестируемы
